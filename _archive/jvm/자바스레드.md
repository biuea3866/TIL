# 자바 스레드

## 스레드
---

스레드는 프로세스보다 가벼운 스케줄링 단위이다. 그리고 메모리(공유 자원)를 공유하며, CPU 자원을 할당받아 작업을 수행할 수 있다. 스레드는 커널 수준(운영체제), 애플리케이션 수준(프로세스), 애플리케이션에서 한 단계 더 추상화(가상 스레드) 등에서 존재하는데 결국 커널 수준의 스레드와 매핑이 되어 동작하게 된다. 

### 커널 스레드

커널 스레드는 OS 커널에서 지원하는 스레드이며, 프로세서와 스레드의 작업을 매핑시켜주는 역할을 한다. 그리고 프로세스(애플리케이션)의 스레드와 일대일로 매핑되어 프로세스의 작업들을 수행하기도 한다. 작업의 단위가 스레드이기 때문에 스레드 하나가 블로킹되는 상황일지라도, 프로세스 전체 작업엔 영향을 끼치지 않는다.

다만, 2가지의 한계점이 존재한다.
1. 커널 기반이기 때문에 스레드의 모든 생명주기가 시스템 콜로 이루어진다. 프로세스는 커널 스레드와 일대일로 매핑되어 작업이 수행되기 때문에 스케줄링에 의한 컨텍스트 스위칭이 발생하는데, 실행 중인 스레드의 레지스터, 스택 등의 상태를 저장하고 다음 스레드의 상태를 복원하는 과정이 커널 모드에서 이루어지므로 오버헤드가 크다.
2. 커널 스레드는 생성 시 커널 스택, TCB(Thread Control Block) 등 OS 자원을 할당받기 때문에 메모리 사용량이 크다. 일반적으로 스레드 하나당 수백 KB~수 MB의 스택 메모리를 차지하므로, 수만 개 이상의 스레드를 동시에 유지하기 어렵다.

### 자바 스레드

자바 스레드는 `java.lang.Thread` 클래스를 통해 생성되는 플랫폼 스레드이다. JVM 내부에서 생성되며, OS의 커널 스레드와 1:1로 매핑된다. 즉, 자바 스레드를 하나 생성하면 OS 커널 스레드도 하나 생성되어 실제 작업을 수행하게 된다.

자바 스레드는 다음과 같은 생명주기(상태)를 가진다.
- **NEW**: 스레드 객체가 생성되었지만, 아직 `start()`가 호출되지 않은 상태
- **RUNNABLE**: `start()`가 호출되어 실행 가능한 상태. OS 스케줄러에 의해 CPU를 할당받으면 실행된다.
- **BLOCKED**: 모니터 락(synchronized)을 획득하기 위해 대기 중인 상태
- **WAITING**: `Object.wait()`, `Thread.join()`, `LockSupport.park()` 등에 의해 다른 스레드의 특정 동작을 무기한 대기하는 상태
- **TIMED_WAITING**: `Thread.sleep(ms)`, `Object.wait(ms)`, `Thread.join(ms)` 등에 의해 지정된 시간만큼 대기하는 상태
- **TERMINATED**: 스레드의 `run()` 메서드가 종료되어 실행이 완료된 상태

자바 스레드는 커널 스레드와 1:1 매핑이므로 커널 스레드의 한계점을 그대로 가진다. 스레드 생성/소멸 비용이 크기 때문에 일반적으로 스레드 풀(ThreadPool)을 사용하여 스레드를 재사용한다. `ExecutorService`와 같은 API를 통해 스레드 풀을 관리할 수 있으며, 이를 통해 스레드 생성 비용을 줄이고 동시 작업 수를 제한할 수 있다.

## 자바 스레드 스케줄링
---

### 선점형 스케줄링

선점형 스케줄링은 OS가 각 스레드에 일정 시간을 할당하고, 그 시간이 지나면 강제로 CPU를 회수하여 다른 스레드에 할당하는 방식이다. 자바는 기본적으로 선점형 스케줄링을 사용한다. JVM이 직접 스케줄링을 하는 것이 아니라, OS 스케줄러에 위임하는 구조이기 때문이다.

자바에서는 `Thread.setPriority()`를 통해 스레드의 우선순위를 1(MIN_PRIORITY)~10(MAX_PRIORITY) 사이로 설정할 수 있다. 하지만 이 우선순위는 OS 스케줄러에 대한 힌트일 뿐, 반드시 보장되지는 않는다. OS마다 우선순위 매핑 방식이 다르고, 스케줄링 정책에 따라 무시될 수 있다.

선점형 스케줄링의 장점은 특정 스레드가 CPU를 독점하는 것을 방지하여 응답성을 보장할 수 있다는 것이다. 반면, 컨텍스트 스위칭이 빈번하게 발생할 수 있어 오버헤드가 존재한다.

### 비선점형 스케줄링

비선점형 스케줄링은 현재 실행 중인 스레드가 자발적으로 CPU를 양보하기 전까지 다른 스레드가 실행되지 않는 방식이다. 스레드가 `Thread.yield()`, `Thread.sleep()`, I/O 대기 등을 통해 자발적으로 실행 흐름을 넘겨야만 다른 스레드가 실행 기회를 얻는다.

자바는 기본적으로 선점형 스케줄링을 사용하지만, `Thread.yield()`를 호출하면 현재 스레드가 자발적으로 CPU 사용 권한을 양보할 수 있다. 다만 이 역시 OS 스케줄러에 대한 힌트이므로, 반드시 다른 스레드가 실행된다는 보장은 없다.

비선점형 스케줄링의 장점은 컨텍스트 스위칭이 적어 오버헤드가 낮다는 것이다. 하지만 하나의 스레드가 CPU를 오래 점유하면 다른 스레드가 기아 상태에 빠질 수 있다는 단점이 있다.

## 가상 스레드와 코루틴
---

### 가상 스레드

가상 스레드는 JVM 수준에서 관리되는 경량 스레드이다. 가상 스레드는 직접 커널 스레드와 매핑되지 않고, 캐리어 스레드(자바 플랫폼 스레드)위에 마운트되어 실행된다. 캐리어 스레드는 기존 자바 스레드와 동일하게 커널 스레드와 1:1로 매핑된다. 즉, 가상 스레드 → 캐리어 스레드(플랫폼 스레드) → 커널 스레드의 구조이며, 수백만 개의 가상 스레드가 소수의 캐리어 스레드 위에서 M:N으로 스케줄링되어 동작할 수 있다.

가상 스레드의 핵심 특징은 다음과 같다.
- **경량성**: 가상 스레드는 수 KB 수준의 스택 메모리만 사용하며, 힙에 저장된다. 따라서 수백만 개의 가상 스레드를 동시에 생성할 수 있다.
- **블로킹 I/O 최적화**: 가상 스레드가 블로킹 I/O를 만나면 캐리어 스레드에서 언마운트되고, 다른 가상 스레드가 해당 캐리어 스레드에 마운트되어 실행된다. 이로 인해 블로킹 I/O가 실제로는 논블로킹처럼 동작한다.
- **기존 코드 호환성**: `Thread` API와 호환되므로 기존 동기 코드를 거의 수정 없이 사용할 수 있다. `Thread.ofVirtual().start(runnable)` 또는 `Executors.newVirtualThreadPerTaskExecutor()`를 통해 생성한다.

다만, CPU 바운드 작업에서는 가상 스레드의 이점이 크지 않다. 결국 CPU 연산은 캐리어 스레드(플랫폼 스레드) 위에서 실행되기 때문이다. 또한 `synchronized` 블록 내에서 블로킹 I/O가 발생하면 캐리어 스레드가 고정(pinning)되어 성능이 저하될 수 있으므로, `ReentrantLock`을 사용하는 것이 권장된다.

### 코루틴

코루틴은 실행을 일시 중단(suspend)하고 재개(resume)할 수 있는 경량 동시성 구조이다. Kotlin의 코루틴이 JVM 생태계에서 가장 대표적이며, 자바의 가상 스레드와 유사한 목적을 가지지만 접근 방식이 다르다.

코루틴의 핵심 특징은 다음과 같다.
- **비선점형 협력적 스케줄링**: 코루틴은 `suspend` 지점에서 자발적으로 실행을 양보한다. OS가 아닌 코루틴 스케줄러(디스패처)가 코루틴의 실행을 관리한다.
- **컴파일러 기반 변환**: Kotlin 컴파일러가 `suspend` 함수를 상태 머신(State Machine)으로 변환한다. 이를 통해 스레드를 블로킹하지 않고도 비동기 코드를 동기 코드처럼 작성할 수 있다.
- **구조적 동시성(Structured Concurrency)**: `CoroutineScope`를 통해 코루틴의 생명주기를 체계적으로 관리한다. 부모 코루틴이 취소되면 자식 코루틴도 함께 취소되어, 리소스 누수를 방지한다.

가상 스레드와 코루틴의 주요 차이점은 다음과 같다.

| 구분 | 가상 스레드 | 코루틴 |
|------|------------|--------|
| 런타임 | JVM 수준에서 관리 | 라이브러리 수준에서 관리 |
| 스케줄링 | 선점형 (OS 스케줄러 위임) | 비선점형 (협력적, suspend 지점에서 양보) |
| 중단 지점 | 블로킹 I/O 호출 시 자동 언마운트 | `suspend` 키워드로 명시적 선언 |
| 코드 스타일 | 기존 동기 코드 그대로 사용 | `suspend`, `async/await` 등 전용 문법 필요 |
