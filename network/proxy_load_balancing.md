# 포워드 프록시, 리버스 프록시, 로드밸런싱

## 포워드 프록시

클라이언트와 인터넷 사이에서 위치하여 클라이언트를 대신하여 프록시 서버가 요청을 보냄

프록시 서버는 중간에서 대리자 역할을 하여 부가적인 기능(보안, 캐싱)등을 처리하여 클라이언트 쪽으로 응답을 밀어주기 때문에 포워드라 불림

* 보안
    * 프록시 서버가 올바른 클라이언트의 요청인지 확인하고, 실제 서버로 요청을 보냄
    * 서버는 실제 클라이언트의 ip를 모름

* 캐싱
    * 이전과 동일한 요청일 경우 효율적인 처리를 위해 캐싱된 응답을 반환

* 컨텐츠 필터
    * 특정 사이트, 컨텐츠에 대한 엑세를 제한할 수 있다

## 리버스 프록시

인터넷과 서버 사이에 위치하여 클라이언트의 요청을 처리하여 서버쪽으로 요청을 밀어주기 때문에 리버스라고 불림

* 로드 밸런싱
    * 여러 서버에 요청들을 균등하게 분산하여 특정 서버가 과한 요청이 몰리지 않도록 함

* 보안
    * 클라이언트는 서버의 ip를 모름

* 캐싱
    * 동일한 이전 요청에 대해 캐싱된 응답을 제공할 수 있다

* 암호화
    * https 요청과 같은 ssl 처리를 프록시 서버에서 담당

## 로드 밸런싱

여러 클라이언트가 보낸 요청을 여러 서버에 균등하게 분산

* 라운드 로빈: 순서대로 돌아가며 요청 분배
* 가중 라운드 로빈: 성능에 따라 가중치 배분
* 최소 연결: 현재 연결 수가 가장 적은 서버 선택
* ip 해시: 클라이언트 ip를 기반으로 고정 서버 할당

### L4 로드 밸런서 (전송 계층)

l4 정보인 ip, port만으로 로드 밸런싱을 수행한다

| source | dest(source) | dest |
| --- | --- | --- |
| client | lb(가상 ip) | server(실제 ip) |

클라이언트는 실제 서버와 커넥션을 맺은 것이 아니고, LB 서버와 커넥션을, LB는 실제 서버와 커넥션을 맺어 세션이 연결된 것처럼 보이고 통신을 LB가 중계한다

전송 계층에서의 라우팅이기 때문에 더 상위의 L7처럼 정보를 래핑할 필요 없이 빠르게 트래픽을 분산시킬 수 있다

L4 정보만으로 분산 처리하기 때문에 헤더, 쿠키, url 기반의 정보로 분산 처리를 할 수 없다

즉, ip, port만으로 요청을 분산시키키 때문에 한계점이 존재한다

### L7 로드 밸런서

클라이언트와 LB의 커넥션, LB와 타겟 서버 간의 커넥션을 맺는다

응용 계층에서 동작하기 때문에 L4에 비해 더 많은 정보 (http, 페이로드)를 이용하여 분산 처리를 할 수 있다

이(url, header, cookie)를 이용하여 분산 처리를 할 수 있으나 L4에 비해 많은 자원을 소모해야하며, 악의적으로 탈취될 경우 보안상의 위협이 생길 수 있다

nginx같은 웹서버로만 L7 로드 밸런서를 구축하나?

spring-cloud-gateway같은 was로 구축이 가능하다 (단, service discovery같이 전역적인 인스턴스를 관리하는 프레임워크와 병행하여 사용해야함)

하지만 역할 분담의 관점에서 web server가 별도로 있는 것이 좋다

* https와 같은 보안 처리
* 정적 콘텐츠 처리

와 같은 역할을 웹서버가 처리하게 하는 것이 좋다
