# WAS, WS

## WS (Web Server)

클라이언트로부터 HTTP 요청을 받고, 정적인 웹 페이지, 이미지, 동영상 등을 반환하는 서버

정적으로 데이터를 서빙하기 때문에 동적인 처리가 불가능 (DB, 사용자 별 페이지 등)하여 CGI가 등장

* 종류
    * Apache HTTP Server
    * Nginx
    * Microsoft IIS
    * Cloudflare
    * ...

### Apache Http Server

* 프로세스 / 스레드 기반 동작
    * 마스터 프로세스 : 워커 스레드 = 1 : n
* 요청당 스레드를 할당하고, 블로킹 io로 동작한다
    * 요청이 많을 경우 메모리, cpu 사용량이 증가하고 요청이 지연된다

### Nginx

* Event Driven 기반 동작
    * 마스터 프로세스 : 워커 프로세스 = 1 : n
* 요청(커넥션)을 이벤트 큐에 담고, 워커 프로세스에게 전달하고, 현재 동작이 가능한 워커 프로세스가 요청을 처리한다
    * 유휴 자원(블로킹으로 인한 대기)없이 효율적으로 자원 사용을 한다

## CGI

웹 서버와 외부 프로그램 간 정보를 주고 받는 인터페이스(규약)

* 요청이 외부 프로그램을 참조해야하면 웹 서버는 fork로 프로세스를 생성
    * 요청이 종료되면 프로세스를 소멸
* 요청 마다 프로세스를 생성하고, 죽이는 작업이기 때문에 메모리, cpu 등 자원 활용에서 비효율적임

## WAS (Web Application Server)

클라이언트의 HTTP 요청에 대해 비즈니스 로직, 동적 컨텐츠를 응답

* 트랜잭션, 복잡한 비즈니스 로직 등을 처리
* 스레드 풀을 활용하여 스레드 단위로 요청을 처리하고, 저비용의 컨텍스트 스위칭을 발생시킴
* 종류
    * Apache Tomcat
    * Oracle WebLogic
    * Jetty
    * IBM WebSphere
    * ...

### 백엔드만 보면 정적인 콘텐츠를 제공하지 않는데 왜 Web Server가 필요할까

웹서버 없이 was만 사용해도 되지만 확장성, 보안성, 부하 분산에서 이점을 가져갈 수 없다

1. **확장성**

앞단의 웹서버를 두고, was만 확장(스케일 아웃)하여 필요 시 많은 요청에 대해 대응할 수 있다

2. **보안성**

https와 같은 암호화/복호화 작업을 웹서버에서 처리해주고, was와는 http로 통신하여 was는 비즈니스 처리에만 집중을 할 수 있게 도와준다

3. **부하 분산**

로드 밸런싱을 사용하여 뒷단의 was서버들에게 골고루 요청을 분산하여 한 곳에서의 과한 요청 처리를 방지한다

### Apache Tomcat

* 웹 서버, 웹 컨테이너의 결합 (서블릿 컨테이너)
* 많이 사용되는 was이고, jsp, 서블릿, http 요청 응답 등을 관리한다

#### 아파치 톰캣

아파치 = Web Server, 톰캣 = Web Application Server인데 아파치 톰캣이라 불리는 이유는 톰캣에서 아파치의 일부 기능을 사용하기 때문이다

하지만 웹 서버의 모든 기능을 제공하지 않기 때문에(로드밸런싱 같은) 따로 웹 서버를 앞에 두어 사용한다
