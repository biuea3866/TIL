# JVM

## 1. JVM이란

JVM은 Java Virtual Machine의 약자이다.
가상 머신이란 이름이 붙은만큼 JVM의 역할은 자바 코드를 가상 머신이 읽고, 동작시키는 데 있다.
그리고 OS에 종속적으로 동작하는 애플리케이션과 달리 자바 애플리케이션은 JVM이라는 중간 다리 역할 위에서 동작하기 때문에 애플리케이션이 OS에 종속적이지 않다.

## 2. JVM의 구조

### 2.1. Java Source

자바의 소스 코드이다.
실제 개발자들이 작성하는 코드로, .java 확장자로 끝나는 파일명으로 구성되어있다.

### 2.2. Java Compiler (javac)

자바 소스 코드를 바이트 코드로 변환하기 위한 자바 컴파일러이다.

### 2.3. Java Byte Code (.class)

자바 컴파일러에 의해 소스 코드가 변환된 바이트 코드이고, .class 확장자로 끝난다.
이 바이트 코드는 JVM위에서 동작하여 실제 자바 애플리케이션이 동작하게 된다.

### 2.4. Class Loader

바이트 코드를 JVM내로 로드하는 역할을 한다.
바이트 코드를 로드(Loading)하고, JVM 명세에 맞는지 검증(Linking)하고, 변수들을 초기화(Initiailzation)하여 JVM의 메모리인 Runtime Data Area로 클래스 파일을 로드한다.

### 2.5. Runtime Data Area

런타임 데이터 영역은 JVM의 메모리 영역으로 자바 애플리케이션이 실행될 때 사용되는 데이터들을 적재하는 영역이다.

#### 2.5.1 메서드 영역

바이트 코드에 정의된 클래스 정보, 변수 정보, 타입 등을 저장하는 공간이다.

#### 2.5.2. 힙 영역

메서드 영역과 함께 모든 스레드가 공유하는 공간이며, 객체들의 생애주기가 관리되는 곳이다.
런타임에서 생성되는 객체들은 메모리에 할당되고, 이 객체들의 멤버 변수, 인스턴스의 주소 값등은 스택 영역에서 관리된다.

#### 2.5.3. 스택 영역

스레드마다 하나씩 가지고 있는 영역이다.
메서드 호출마다 스택 프레임이 만들어지고, 현재 실행중인 메서드의 상태 정보를 저장한다.
이 스택 프레임에서는 매개 변수, 지역 변수, 리턴 값등이 존재한다.

### 2.6 Execution Engine

Runtime Data Area에 배치된 바이트 코드를 명령어 단위(컴파일된 행)로 읽어서 실행하는 역할을 한다.
실행엔진은 인터프리터와 JIT 컴파일러 방식을 혼합하여 바이트 코드를 실행한다.

#### 2.6.1 인터프리터

바이트 코드를 하나씩 읽고, 실행한다.
기본적으로 인터프리터 방식으로 동작한다.

#### 2.6.2 JIT 컴파일러

인터프리터는 반복되는 호출 패턴을 가진 바이트코드일지라도, 매번 해석하고 수행하는 과정을 가진다.
JIT 컴파일러는 이런 핫코드들을 감지하여 네이티브 코드로 캐싱해두었다가 네이티브 코드(기계어)를 바로 실행하는 방식으로 인터프리터보다 훨씬 빠른 성능을 가지고 있다.
다만, 모든 바이트코드를 캐싱할 순 없으므로 일정 기준치가 넘어가는 바이트 코드를 JIT 컴파일러가 캐싱한다.

#### 2.6.3 가비지 컬렉터

더 이상 참조되지 않는 객체들을 수거하는 역할을 한다.
힙에는 eden, survivor(1, 2), old, permanent 등의 영역이 존재하고, 얼마나 참조(age)되었는지 그리고 객체의 크기에 따라 각 영역으로 보내지고 더 이상 참조의 대상이 아닌 객체들은 수거 대상이 된다.

## 3. JVM의 동작

1. 자바 애플리케이션을 실행하면, JVM은 운영체제로부터 메모리를 할당받는다.
2. 자바 컴파일러는 자바 소스코드를 자바 바이트 코드로 변환한다.
3. 클래스 로더는 이 바이트 코드를 런타임 메서드 영역에 로드한다.
4. Execution Engine은 이 바이트 코드를 읽고, 기계어로 변환하고 실행한다.

# JDK, JRE, JVM 관계

```
JDK (Java Development Kit)
├── 개발 도구 (javac, javadoc, jar, jdb ...)
└── JRE (Java Runtime Environment)
    ├── 라이브러리 API (java.lang, java.util, java.io ...)
    └── JVM (Java Virtual Machine)
        ├── Class Loader
        ├── Runtime Data Area (메서드, 힙, 스택 ...)
        └── Execution Engine (인터프리터, JIT, GC)
```

# JRE

Java Runtime Environment의 약자로 JVM, 자바 프로그램을 실행시킬 때 필요한 라이브러리 API를 함께 묶어 배포하는 패키지이다.

# JDK

Java Development Kit의 약자로 javac, javadoc 등의 개발도구가 포함되어 있다.

## JDK 버전

JDK의 버전은 주버전, 개선버전, 업데이트버전, 장기 지원 서비스 버전으로 구성되어있다.

```
Java SE 11.0.2(LTS)
```

오라클 및 다른 벤더사들은 특정 주기로 LTS 버전을 내놓는데(오라클은 2년마다) 이 사이 동안 릴리즈된 버전은 시험용 버전이다.
즉, 이 시험용 피쳐들의 안정성과 보안성을 검증하여 특정 주기로 LTS 버전을 내놓게 된다.

## JDK 종류

* Oracle JDK: 오라클에서 제공하는 JDK로 최신의 기능과 패치가 적용되고 보안과 안정성에서 좋지만, 유료 플랜이다.
* OpenJDK: 무료 JDK이나 LTS가 6개월로 짧기 때문에 안정성과 보안을 보장받을 수 없다.
* Temurin: Red Hat 등 기업이 참여하면서 무료 JDK이다.
* Azul Zulu: 다양한 플랫폼(구형 OS, ARM등)을 지원하는 무료 JDK이다.
* Correctto: 아마존이 지원하는 JDK로 AWS의 수 많은 서비스에서 검증되었다.

## JDK 구성요소

* bin: 자바 개발, 실행에 필요한 도구와 유틸리티 명령
    * javac: 자바 컴파일러로 소스 코드를 바이트 코드로 컴파일하는 역할
    * java: 자바 인터프리터로 바이트 코드를 해석하고 실행하는 역할
    * javadoc: 자바 소스 코드를 HTML 형식의 API 도큐먼트로 생성한다.
    * jar: 자바 클래스 파일을 압축한 아카이브 파일
    * jdb: 자바 디버깅 툴
* include: 네이티브 코드인 C언어 헤더 파일
* lib: 런타임에 필요한 라이브러리 클래스

# JVM에서의 코틀린

코틀린은 Java 기반의 언어가 아닌 JVM 기반의 언어이다.
다만, Java 언어와의 호환성을 자랑할 뿐이지, Kotlin → Java → 바이트 코드로의 변환을 의미하지 않는다.
코틀린은 코틀린 컴파일러가 독자적으로 존재하고, 이 컴파일러가 코틀린 소스코드를 바이트 코드로 변환하여 JVM에 로드되는 형태이다.
즉, Java가 JVM로드되는 흐름과 동일하다.
