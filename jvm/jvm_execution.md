# [JVM] 자바 실행 흐름

## 1. 자바와 JVM

---

### 1.1 JVM의 탄생 배경

C/C++ 같은 언어로 작성된 소프트웨어는 OS에 종속적이다. 동일한 소스 코드라도 OS마다 컴파일을 따로 해야 했고, 그 결과물(실행 파일)을 이용하여 소프트웨어를 실행했다.

이런 OS 종속성 문제를 해결하기 위해 **Write Once, Run Anywhere**라는 철학으로 자바가 만들어졌다.

![](./img/compiler/compiler_1.png)

### 1.2 JVM이란

JVM은 **Java Virtual Machine**의 줄임말로 자바 코드를 동작시키기 위한 가상 머신이다.

가상 머신은 물리 컴퓨터 위에 소프트웨어로 만든 독립적인 가상의 컴퓨터 환경이며, JVM은 자바 소스 코드를 컴파일하여 만든 바이트 코드를 실행시키기 위한 가상 머신이다.

JVM은 OS 위에서 독립적으로 동작하기 때문에, JVM이 동작 가능한 OS라면 어디서든 자바 애플리케이션을 실행할 수 있다.

#### .exe와의 비교

C/C++의 경우 개발자는 .cpp 파일을 작성하고, 컴파일하여 .exe라는 실행 파일을 생성한다. 이 .exe는 OS에 종속적인 결과물이라 다른 OS에서는 동작하지 않는다.

![](./img/jvm_exec_1.png)

반면 자바는 OS 위에 래핑된 형태로 동작하는 JVM이라는 가상 머신을 한 단계 더 두었기 때문에, 동작 가능한 OS라면 어디서든 실행이 가능하다.

### 1.3 JDK / JRE / JVM 관계

```
JDK (Java Development Kit)
├── 개발 도구 (javac, javadoc, jar, jdb ...)
└── JRE (Java Runtime Environment)
    ├── 라이브러리 API (java.lang, java.util, java.io ...)
    └── JVM (Java Virtual Machine)
        ├── Class Loader
        ├── Runtime Data Area (메서드, 힙, 스택 ...)
        └── Execution Engine (인터프리터, JIT, GC)
```

- **JVM**: 바이트 코드를 실행하는 가상 머신.
- **JRE**: JVM + 자바 프로그램 실행에 필요한 라이브러리 API를 묶은 패키지.
- **JDK**: JRE + javac, javadoc, jdb 같은 개발 도구를 포함.

#### JDK 구성 요소

- `bin`: 자바 개발·실행에 필요한 도구
    - `javac`: 자바 컴파일러로 소스 코드를 바이트 코드로 컴파일
    - `java`: 자바 인터프리터로 바이트 코드를 실행
    - `javadoc`: 소스 코드를 HTML API 도큐먼트로 생성
    - `jar`: 자바 클래스 파일을 압축한 아카이브 파일
    - `jdb`: 자바 디버깅 툴
- `include`: 네이티브 코드용 C 헤더 파일
- `lib`: 런타임에 필요한 라이브러리 클래스

### 1.4 JDK 종류와 버전

JDK 버전은 주버전·개선버전·업데이트버전·LTS 여부로 구성된다.

```
Java SE 11.0.2 (LTS)
```

오라클을 비롯한 벤더사들은 정해진 주기(오라클 기준 2년)마다 LTS 버전을 내놓는다. 그 사이 6개월 단위로 릴리즈되는 비-LTS 버전은 새 기능을 검증하는 시험용 버전 성격을 갖는다.

#### JDK 종류

| JDK | 특징 |
|-----|------|
| **Oracle JDK** | 오라클이 제공. 최신 기능과 보안 패치가 빠르며 상용 환경 지원이 좋지만 유료 플랜 |
| **OpenJDK** | 무료 JDK. 6개월 주기로 새 버전 출시, 일부 버전(11/17/21 ...)이 LTS로 지정. 단, 벤더별로 LTS 패치 지원 기간이 다름 |
| **Eclipse Temurin** | Eclipse Adoptium 프로젝트가 빌드·배포하는 무료 JDK. 광범위한 플랫폼 지원 |
| **Azul Zulu** | 다양한 플랫폼(구형 OS, ARM 등)을 지원하는 무료 JDK |
| **Amazon Corretto** | 아마존이 지원하는 무료 JDK. AWS의 수많은 서비스에서 검증됨 |

### 1.5 JVM 위의 코틀린

코틀린은 자바 기반의 언어가 아닌 **JVM 기반의 언어**이다. 자바 언어와의 호환성을 가지긴 하지만, Kotlin → Java → 바이트 코드 변환은 아니다.

```
.kt  ─kotlinc→  .class  ─ClassLoader→  JVM 위에서 실행
.java ─javac→  .class  ─ClassLoader→  JVM 위에서 실행
```

코틀린 컴파일러(`kotlinc`)가 독자적으로 존재하며, 코틀린 소스 코드를 바이트 코드로 직접 변환하여 JVM에 로드한다. JVM에 올라간 이후의 흐름은 자바와 동일하다.

## 2. 컴파일: 소스 코드 → 바이트 코드 (프론트엔드 컴파일)

---

### 2.1 javac의 역할

`javac`는 **프론트엔드 컴파일러**로, JVM이 플랫폼 독립적으로 운영되기 위해 자바 코드를 중간 언어인 바이트 코드로 변환하는 역할을 한다.

프론트엔드 컴파일은 코드 최적화보다는 다음 사항을 검사한다.

- 자바 코드가 문법적으로 문제는 없는가
- 변수, 메서드, 인스턴스가 올바른 위치에 있는가

이 검사를 위해 구문 검사, 분석, 추상 구문 트리(AST) 구성 등의 과정을 거친다. 실제 코드 최적화는 런타임의 백엔드 컴파일(JIT, 5.2 참조)이 담당한다.

![](./img/compiler/compiler_2.png)

### 2.2 컴파일 단계

#### 1) 구문 분석과 심벌 테이블 채우기

**어휘 분석**: 소스 코드의 문자 스트림을 토큰 집합(키워드, 변수명, 리터럴, 연산자)으로 변환한다.

```kotlin
int x = 10; // [int][x][=][10][;]
```

**구문 분석**: 토큰들로부터 추상 구문 트리(AST)를 구성하고, 문법 규칙에 맞는지 검사한다.

**심벌 테이블 채우기**: 심벌 테이블에 심벌 주소와 심벌 정보 집합을 구성하여 중간 코드 생성·목적 코드 생성 단계에 활용한다.

> **심벌 테이블**: 변수 관리를 위해 `key : value = 변수 : 메모리 주소`로 관리되는 테이블

#### 2) 애너테이션 처리

특정 애너테이션이 컴파일 과정에서 추상 구문 트리를 변경시킬 수 있다. 변경이 발생하면 1번으로 되돌아가 더 이상 AST를 변경시킬 수 없을 때까지 반복한다.

#### 3) 의미 분석과 바이트 코드 생성

**특성 검사**: 사용된 변수가 이미 선언되었는지, 타입은 일치하는지를 확인한다.

**데이터·제어 흐름 분석**: 맥락상 코드가 논리적으로 올바른지 확인한다. 가령 선언되지 않은 변수가 계산식에 포함되어 있는 경우 등이다.

**편의 문법 제거**: 제네릭, 가변 길이 매개변수, 오토박싱·언박싱 등은 런타임에 직접 지원하지 않는다. 따라서 기본 문법으로 전환하여 컴파일을 진행한다.

**바이트 코드 생성**: 바이트 코드 명령어로 변환하여 저장소에 기록한다.

### 2.3 바이트 코드

바이트 코드는 JVM이 이해할 수 있는 중간 언어로, OS와 CPU에 종속적이지 않다. 어느 OS에서든 동일한 바이트 코드를 갖고, JVM만 설치 가능하다면 자바 애플리케이션을 실행할 수 있다.

```kotlin
// Java 소스
public int add(int a, int b) {
    return a + b;
}

// 바이트 코드 (javap -c MyClass.class)
public int add(int, int);
  Code:
     0: iload_1   // 로컬 변수 a를 피연산자 스택에 push
     1: iload_2   // 로컬 변수 b를 피연산자 스택에 push
     2: iadd      // 스택 상위 2개를 꺼내 더하고 결과를 push
     3: ireturn   // 스택 상위 값을 반환
```

바이트 코드는 JVM의 인터프리터 또는 JIT 컴파일러가 읽어가며 실행된다.

## 3. 클래스 로딩: 바이트 코드 → JVM

---

### 3.1 클래스 로더의 역할

클래스 로더는 자바 바이트 코드(.class)를 JVM에 로드하여 Runtime Data Area에 배치하는 역할을 한다.

#### 동적 로딩

클래스 로더는 동적으로 로드한다. 프로그램 시작 시 모든 클래스를 한 번에 로드하는 게 아니라, **필요한 시점**에 클래스를 로드한다.

자바는 인터페이스·다형성을 제공하고 리플렉션 API를 제공하기 때문에 컴파일 시점에 구체 타입이나 동적 클래스를 알 수 없다. 그래서 클래스 로더는 필요한 시점에 클래스 정보를 로드하는 동적 방식을 취한다.

### 3.2 클래스 로더 종류

JDK 9 이후 기준으로 다음 3계층 구조를 갖는다.

| 종류 | 역할 |
|-----|------|
| **Bootstrap ClassLoader** | JVM 핵심 클래스(`java.lang.*`, `java.util.*` 등)를 로드. 네이티브 코드로 구현 |
| **Platform ClassLoader** (Java 9+) | 플랫폼 모듈(JCE, JAXP 등)을 로드. Java 8에서는 Extension ClassLoader |
| **System(Application) ClassLoader** | 애플리케이션 classpath의 클래스를 로드 |

### 3.3 부모 위임 모델 (Parent Delegation Model)

클래스 로더는 클래스를 로드해달라는 요청이 오면, **자기가 로드하기 전에 먼저 부모 로더에게 위임**한다.

```
요청: java.lang.String 로드
  System ClassLoader → "부모에게 먼저 물어봄"
    Platform ClassLoader → "부모에게 먼저 물어봄"
      Bootstrap ClassLoader → "내가 가지고 있다, 로드해서 반환"
```

이 위임 모델 덕분에 `java.lang.String` 같은 핵심 클래스는 항상 Bootstrap이 로드하고, 사용자가 같은 이름의 악성 `String` 클래스를 만들어도 위에서 가로채진다. 즉, **클래스 식별의 일관성과 보안**을 보장한다.

### 3.4 로딩 단계

클래스 로더는 다음 단계를 거쳐 클래스를 로드한다.

#### 1) Loading

바이트 코드를 메서드 영역에 저장한다. 저장하는 정보는 다음과 같다.

- 로드된 클래스와 부모 클래스의 정보
- 클래스, 인터페이스, Enum 클래스 정보
- 변수, 메서드 정보

#### 2) Linking

로딩된 정보가 JVM 명세에 맞게 구성되었는지 검사하고, 정적 필드를 위한 메모리를 할당한다. 세 단계로 나뉜다.

- **Verify**: 바이트 코드가 JVM 명세를 위반하지 않는지 검사 (스택 오버플로우, 잘못된 타입 변환 등)
- **Prepare**: 클래스의 정적 필드를 기본값(0, null, false 등)으로 초기화한 메모리 할당
- **Resolve**: 심벌 참조(예: `java/lang/String` 같은 클래스 참조)를 직접 참조(메모리 주소)로 변환

#### 3) Initialization

`static` 블록을 실행하고, 정적 필드를 사용자가 명시한 값으로 초기화한다. (Prepare는 기본값, Initialization은 명시값)

## 4. JVM 메모리 구조 (런타임 데이터 영역)

---

![](./img/jvm_exec_2.png)

런타임 데이터 영역은 자바 애플리케이션이 실행되는 동안 클래스 정보·변수·객체 등을 저장하고 사용하는 메모리 공간이다. 메서드 영역, 힙, PC 레지스터, 가상머신 스택, 네이티브 메서드 스택으로 구성된다.

영역은 크게 두 부류로 나뉜다.

| 분류 | 영역 |
|-----|------|
| 모든 스레드가 공유 | 메서드 영역, 힙 |
| 스레드 독립적 | 가상머신 스택, PC 레지스터, 네이티브 메서드 스택 |

### 4.1 공유 영역

모든 스레드가 접근할 수 있다. 공유 영역이라는 것은 동기화 문제를 신경 써야 하는 지점이라는 뜻이기도 하다.

#### 메서드 영역

클래스 정보, 상수, 바이트 코드, JIT 컴파일러가 컴파일한 코드 캐시 등이 저장된다. 클래스 로더가 변환된 바이트 코드를 JVM으로 로드하면, 그 바이트 코드를 메서드 영역에 저장하게 된다.

이 영역의 일부로 **런타임 상수 풀**이 존재한다. 이곳엔 클래스의 메타정보와 심벌 참조(예: `java/lang/String` 같은 클래스 참조) 등을 저장한다.

> Java 8부터는 클래스 메타데이터를 관리하던 힙의 PermGen이 제거되고, 네이티브 메모리의 **Metaspace**로 대체되었다.

#### 힙 영역

객체들을 저장하는 공간이다. 힙 영역에서 객체의 생애주기가 다뤄지며, 런타임에 객체가 힙 영역의 메모리를 할당받고, 더 이상 참조가 끊기면 GC 대상이 되어 객체가 소멸된다.

### 4.2 스레드 독립 영역

#### 가상머신 스택

스레드는 메서드를 호출하여 명령을 수행한다. 메서드 호출마다 **스택 프레임**이 만들어지고, 스택 프레임에는 지역 변수 테이블·피연산자 스택·동적 링크·리턴 값이 쌓인다.

스택 프레임을 가상 머신 스택에 쌓고, 메서드 실행이 끝나면 꺼내는 과정을 반복한다. 스레드가 독립적으로 명령을 수행하기 때문에 다른 스레드가 접근해선 안 되므로 스레드 내에서 관리된다.

#### PC 레지스터

> 한 줄 정의: **스레드별로 다음 실행할 바이트 코드 주소를 가리키는 포인터.**

멀티스레딩 환경에서는 OS 스케줄링에 의해 스레드가 CPU 코어를 번갈아 점유한다. 스레드는 작업이 끝나기 전에 잠시 멈추는 상황이 발생하고, 다음 스케줄링 때 작업을 이어서 진행한다.

이때 멈춘 지점이 어디엔가 관리되어야 다음 스케줄링 때 작업을 이어서 진행할 수 있다. 이 "어디엔가"가 PC 레지스터이며, 각 스레드가 독립적으로 가진다.

#### 네이티브 메서드 스택

자바 이외의 언어로 작성된 네이티브 코드를 호출할 때 사용되는 영역이다. 네이티브 메서드 호출 시 생성되며, 가상 머신 스택과 별도로 구현된 JVM도 있고 합친 형태로 구현된 JVM도 있다.

## 5. 실행 엔진: 바이트 코드 → 기계어

---

런타임 데이터 영역에 배치된 바이트 코드를 명령어 단위로 읽어서 실행하는 역할이다. 실행 엔진은 인터프리터와 JIT 컴파일러를 혼합하여 바이트 코드를 실행한다.

### 5.1 인터프리터

인터프리터는 컴파일된 바이트 코드를 한 줄씩 읽어 기계어로 번역하고 명령을 수행한다. 컴파일된 바이트 코드는 JVM에 올릴 수 있는 중간 단계일 뿐, 실제로 실행된 것은 아니다.

#### 한계

인터프리터는 빠르게 실행을 시작할 수 있지만, 한 줄씩 읽기 때문에 다른 컴파일러 언어(소스 코드 전체를 미리 기계어로 컴파일하여 통째로 수행하는 방식)에 비하면 속도가 현저히 느릴 수밖에 없다. 같은 코드가 반복적으로 호출되어도 매번 해석하는 비용을 다시 치른다.

### 5.2 JIT 컴파일러 (백엔드 컴파일)

![](./img/compiler/compiler_3.png)

#### 등장 배경

순수 인터프리터로 한 줄씩 읽었을 때의 느린 성능 문제를 개선하기 위해 등장한 것이 백엔드 컴파일이다.

백엔드 컴파일은 자주 수행되는 코드(**핫코드**)를 백그라운드에서 네이티브 코드로 컴파일하여 미리 준비해두고, 다음 실행 때 더 빠르게 코드를 실행시키는 기법이다.

JIT 컴파일러는 런타임에 바이트 코드를 네이티브 코드로 변환하는 전략이며, C1·C2 (또는 C2의 대안인 Graal) 등이 모두 JIT 컴파일러에 해당한다.

JIT가 정적 컴파일(AOT)보다 좋은 점은 자주 쓰이는 코드의 빈도수를 알고 있다는 것이다. AOT는 미리 컴파일하기 때문에 어떤 코드가 자주 쓰이는지 모르지만, JIT는 통계를 기반으로 필요한 코드들에 대해서만 효율적으로 컴파일을 수행한다.

#### 핫코드와 카운터

JIT 컴파일러의 대상은 핫코드(자주 호출되는 메서드, 순환문 본문을 포함하는 메서드)이다. 어떤 코드가 핫코드인지 판별하기 위해 두 가지 카운터를 사용한다.

##### 메서드 호출 카운터

메서드가 얼마나 호출됐는가를 기록한다. `-XX:CompileThreshold`로 임계치를 지정할 수 있고, 임계치를 넘어서면 JIT 컴파일러는 백그라운드에서 비동기로 컴파일을 수행한다.

##### 백 에지 카운터 (Back Edge Counter)

> **백 에지(back edge)**: 반복문 끝에서 시작 지점으로 돌아가는 분기. 이 분기가 임계치 이상 반복되면 해당 반복문은 핫코드로 간주된다.

백 에지 카운터는 순환문의 코드가 얼마나 반복되는지를 계산한다. 순환문 코드는 메서드에 포함되어 있기 때문에, 스택 프레임에서 실행 중인 이 메서드를 도중에 변환하는 작업을 **OSR(On-Stack Replacement, 온스택 치환)** 컴파일이라 한다.

OSR 덕분에 인터프리터로 한 줄씩 계속 반복하지 않고, 실행 도중에 컴파일된 네이티브 코드로 갈아끼울 수 있다.

```
인터프리터로 반복 중 반복 횟수↑ → OSR 트리거 → 네이티브 코드로 교체 → 같은 반복문이 빠르게 실행
```

#### C1 컴파일러

C1은 빠른 컴파일 속도가 강점이지만, 코드 최적화 면에서는 C2보다 떨어진다. 인터프리터와 달리 네이티브 코드로 컴파일을 수행하며 프로파일 데이터도 수집한다.

```kotlin
// 1. 메서드 인라이닝 (Inlining)
//    메서드 호출 오버헤드 제거
int result = square(x);         // 원본: 메서드 호출
private int square(int n) { return n * n; }

// → C1 최적화 후
int result = x * x;             // 본문이 직접 삽입됨

// 2. Dead Code Elimination
if (false) {
    heavyOperation();           // 실행 불가 코드 제거
}

// 3. 기본 상수 전파 (Constant Propagation)
int x = 10;
int y = x + 5;                  // → int y = 15;
```

#### C2 컴파일러

C2는 컴파일 속도는 느리지만, 처리량 부문에서 가장 좋은 성능을 보인다. C1에서 수집한 프로파일 데이터를 이용하여 핫코드를 깊게 최적화한다.

```kotlin
// 1. 탈출 분석 (Escape Analysis)
//    객체가 메서드 밖으로 탈출하지 않으면 힙 대신 스택에 할당
public int computeArea() {
    Point p = new Point(3, 4);  // p는 이 메서드 밖으로 탈출하지 않음
    return p.x * p.y;
}
// → new Point() 힙 할당 제거, GC 부담 없음

// 2. 루프 언롤링 (Loop Unrolling)
for (int i = 0; i < 4; i++) {
    data[i] = 0;
}
// → data[0]=0; data[1]=0; data[2]=0; data[3]=0;
//    분기 횟수를 줄여 CPU 파이프라인 효율 향상

// 3. 추측적 최적화 (Speculative Optimization)
//    프로파일링 데이터 기반으로 "아마도 이럴 것"이라고 가정하고 최적화
//    가정이 깨지면 → Deoptimization (인터프리터로 복귀 후 재컴파일)
shape.draw();  // 항상 Circle.draw() 였다면
// → if (shape is Circle) Circle.draw(); // 가상 메서드 디스패치 제거
//   else deoptimize;

// 4. 강도 감소 (Strength Reduction)
x * 2   →   x << 1    // 곱셈 → 비트 시프트 (더 빠름)
x * 8   →   x << 3
```

#### 계층형 컴파일 (Tiered Compilation)

C1은 빠른 응답이 강점이고, C2는 느리지만 성능 면에서 우세하다. 둘의 장점을 함께 쓰기 위해 등장한 것이 **계층형 컴파일**이다. 인터프리터부터 C2까지 단계를 두어, 코드 실행 빈도에 따라 점점 더 깊게 최적화한다.

![](./img/compiler/compiler_4.png)

| 계층 | 동작 |
|-----|------|
| **계층 0** | 인터프리터 레벨에서 순수 해석 |
| **계층 1** | 단순 메서드나 빠르게 컴파일 가능한 코드를 C1으로 컴파일·실행 |
| **계층 2** | C1 사용 + 메서드/반환 횟수 등 기본 모니터링 수행 |
| **계층 3** | C1 사용 + 모든 성능 모니터링 수집 |
| **계층 4** | C2 사용 + 수집한 모니터링 정보를 활용한 최적화 + 네이티브 컴파일 |

#### Code Cache

![](./img/compiler/compiler_5.png)

컴파일된 네이티브 코드는 **Code Cache**라는 공간에 저장된다. Java 8 이후 Code Cache는 JVM 내 별도 네이티브 메모리 영역에 위치한다(메서드 영역과 분리됨).

Code Cache가 가득 차면 JIT 컴파일이 중단되기 때문에, 필요 시 `-XX:ReservedCodeCacheSize`로 크기를 조절할 수 있다.

#### 역최적화 (Deoptimization)

JIT 컴파일러는 프로파일 데이터를 기반으로 낙관적 최적화(추측 최적화)를 수행한다. 가정이 깨지면(예: `shape`가 항상 `Circle`이라 가정했는데 다른 타입이 들어옴) 해당 컴파일 결과를 폐기하고 인터프리터 모드로 되돌아가서 다시 계층형 컴파일을 수행한다.

### 5.3 AOT 컴파일러

AOT(Ahead-Of-Time)는 동적으로 네이티브 코드를 컴파일하는 JIT와 달리, **미리** 코드를 네이티브 코드로 만들어 놓고 프로그램을 실행하는 방식이다.

| 항목 | JIT | AOT |
|-----|-----|-----|
| 컴파일 시점 | 런타임 (백그라운드) | 빌드/배포 시점 |
| 시작 속도 | 느림 (워밍업 필요) | 빠름 (즉시 네이티브 실행) |
| 정점 성능 | 높음 (프로파일 기반 최적화) | 보통 (런타임 정보 없음) |
| 메모리 | 높음 (Code Cache 등) | 낮음 |
| 대표 사례 | HotSpot JVM | GraalVM Native Image |

AOT는 실행 시점엔 미리 만들어둔 네이티브 코드가 수행되기 때문에 JIT의 워밍업 단점이 없다. 다만 전체 소스 코드를 네이티브로 컴파일하기 전까지 프로그램이 실행될 수 없으며, 컴파일 시간이 길고, 낙관적 컴파일 결과 역최적화가 필요할 경우 미리 만들어둔 컴파일 코드를 폐기해야 할 수도 있다.

### 5.4 가비지 컬렉터 (개요)

더 이상 참조되지 않는 객체를 메모리에서 해제하여 JVM의 힙 공간을 관리한다. Young, Survivor 0/1, Old 등의 공간이 나누어진 힙에 대해서 가비지 컬렉팅을 수행한다.

- **Young (Eden + Survivor 0/1)**: Minor GC가 발생하는 영역. 대다수 객체가 여기서 해제됨. Young 영역은 작기 때문에 처리할 공간이 작아 빠르게 수행됨.
- **Survivor**: Minor GC에서 살아남은(아직 참조 중인) 객체들이 이동하는 공간. Mark and Copy를 통해 살아남을 때마다 객체 헤더의 age 값이 증가하고, Survivor 0과 1을 왔다갔다한다. age가 임계치에 도달하면 Old 영역으로 promotion한다.
- **Old**: 오래 살아남고 메모리 공간을 많이 차지하는 객체들이 존재. Old 영역에서 발생하는 Major GC는 Minor GC보다 시간이 오래 걸린다.

> 자세한 내용은 [garbage_collector.md](./garbage_collector.md) 참조.

## 6. 런타임 동작

---

### 6.1 객체 생성 흐름

애플리케이션이 동작하면서 외부로부터 요청이 들어온다. 이 요청은 런타임에서 동적으로 수행되고, JVM은 객체를 생성하라는 바이트 코드를 만나 다음 순서로 객체를 생성한다.

1. `java.lang.String`과 같은 클래스 이름(심벌 참조)인지 확인한다.
2. 이 심벌 참조가 클래스 로더에 의해 로딩·링크·초기화 과정을 거쳐 JVM에 로드되었는지 확인한다. (로드되지 않았다면 클래스 로딩이 필요하다.)
3. 객체에 필요한 메모리 크기를 계산하여 자바 힙의 메모리 블록을 할당한다.
4. 생성자를 통해 필드의 값을 초기화하면 객체가 완성된다.

#### 동시성 문제

자바 힙은 공유 공간이기 때문에 메모리 블록 할당 과정에서 동시성 문제가 발생할 수 있다. 이를 해결하기 위해 다음 방식을 사용한다.

- **CAS (Compare-And-Swap) 같은 Atomic 연산**: 충돌이 적을 때 효율적
- **TLAB (Thread-Local Allocation Buffer)**: 스레드별로 사전에 할당받은 작은 힙 영역에서 객체를 할당. 락 없이 빠르게 할당 가능

### 6.2 객체 메모리 레이아웃

객체는 자바 힙에 메모리를 할당받을 때 **헤더 / 인스턴스 데이터 / 정렬 패딩**으로 구성된 메모리 레이아웃 형태로 저장된다.

#### 객체 헤더

- **마크 워드(Mark Word)**: 해시 코드, GC age, 락 플래그 정보를 저장. 32비트 또는 64비트.
- **클래스 워드(Class Pointer)**: 메타스페이스에 저장된 클래스 정보를 가리키는 포인터. 런타임에 어느 클래스의 인스턴스인지 알 수 있게 해 준다.

> HotSpot JVM 기준 헤더 전체 크기는 **압축 OOPs(Compressed OOPs) 사용 시 12바이트, 미사용 시 16바이트**다. 마크 워드만 32/64비트 고정이 아니라, 헤더 전체를 다른 단위로 봐야 한다.

#### 인스턴스 데이터

객체의 실제 정보다. 가령 `class Apple(val color: String)`이라는 클래스가 존재했을 때 `color`의 실제 값이 저장되고, 상속받은 부모 클래스의 필드들도 저장된다.

#### 정렬 패딩

모든 객체는 8바이트 배수로 정렬되어야 한다. 인스턴스 데이터가 8바이트 배수가 되지 않을 때 채워 넣는 영역이다.
