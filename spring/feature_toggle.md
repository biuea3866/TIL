# 피쳐 토글

## 개요

매 스프린트마다 사용중인 API에 추가적인 기능, 로직 수정 등으로 api의 동작을 변화시킵니다.

그리고 변화된 코드는 테스트를 거치고 dev -> prod 까지 배포가 진행됩니다.

이 과정에서는 다음과 같은 우려 포인트들이 생깁니다.

* 롤백 시엔 어떻게 대처하는가
    * 해당 코드 외에 다른 변경점들까지 함께 롤백이 되어야하는가
* 긴 스프린트 속에서 만들어진 모든 코드들이 한번에 머지된다면 다른 작업자들은 컨플릭을 어떻게 감당해야하는가
* 매번 엔드포인트를 버전을 올리면서 복사 붙여넣기를 반복해야하는가

등 외에도 많은 우려 포인트들이 존재할 것입니다.

## 이제까지 어떤 방법들을 사용해왔나

### 복사 붙여넣기 식의 v2 api

```kotlin
@PostMapping(value = ["/v1"])
fun v1Api() {
    v1Service.execute()
}

@PostMapping(value = ["/v2"])
fun v2Api() {
    v2Service.execute()
}
```

이제까지의 작업은 v1 코드를 복사하여, v2로 만들면서 신규 로직을 추가하는 방식으로 진행되었습니다.

초기엔 코드도 복잡하지 않고, 빠르게 처리할 수 있었고, 바로바로 배포되어도 라이브 환경에서 아무 영향도 주지 않는 작업이었기에 상당히 편리한 이점이 있었습니다.

하지만 비즈니스 로직이 복잡해지고, 그에 따라 하나의 api에 상당히 많은 코드가 생성될수록 이 방법은 사용하기가 어려워졌습니다.

하나의 pr에 1000줄이 넘어가는 경우도 많아졌기 때문입니다.

신규 피쳐를 작업할수록 리뷰를 하는 사람도, 코드를 작성하는 사람도 전부 지쳐가는 형태가 되었습니다.

### 버전업된 v2 api와 분기 처리

복사 붙여넣기가 아닌 api 버전을 올리되, 내부에서 변화하는 로직에만 분기를 만들어 코드를 작성하는 방식을 적용하게 되었습니다.

```kotlin
@PostMapping(value = ["/v1"])
fun v1Api() {
    service.execute(version = "v1")
}

@PostMapping(value = ["/v2"])
fun v2Api() {
    service.execute(version = "v2")
}

fun execute(version: String) {
    something()

    if (version == "v2") somethingV2()
}
```

이와 같은 구성은 공통 서비스 내부에서 분기를 이용하기 때문에 코드 중복 감소, 즉각적인 배포 가능 등 다양한 이점을 가져다 주었습니다.

특히 한번에 많은 코드를 배포하지 않아도 된다는 점에서 상당히 편한 작업을 이어갈 수 있었습니다.

하지만 이 방식 또한 문제가 있었는데 피쳐 때 사용할 모든 api를 버전을 올려주어야 한다라는 단점이었습니다.

만약 어떤 로직이 상당히 많은 api가 참조하고 있고, 이를 수정해야한다면 모든 api에 대한 버전을 관리해주어야했고, 현실적으로 모든 api에 대한 버전 관리는 큰 무리가 있습니다.

변천사를 거듭하며, 좀 더 좋은 방법은 없을까하는 고민 중에 피쳐 플래그라는 방법에 대해 알게 되었고, 이 고민들에 대한 좋은 방법론이 될 것이라 생각이 들었습니다.

## Feature Flag

### 이제까지와 무엇이 다른가

가장 큰 점은 런타임에 동적으로 원하는 분기를 이용할 수 있다는 장점이 존재합니다.

이제까지 사용해 온 방법은 결국엔 api 버저닝을 통해 클라이언트에서 api를 호출하게 하며, 정적으로 원하는 기능을 사용하게 되었습니다.

이는 결국 클라이언트의 변경까지 강제해야한다는 면에서 2배의 수고가 필요한 면이 존재했습니다.

피쳐 플래그는 정적으로 코드를 컨트롤하지 않아도 되며, 원하는 시점에 원하는 코드를 사용할 수 있고, 호출부의 작업만으로 안정적인 피쳐를 배포할 수 있는 장점이 존재합니다.

원하는 feature의 기능을 키고 끔에 따라 Service에서는 그에 맞는 로직을 수행하도록 동작하게 하는 방법입니다.

### 어떻게 만들 수 있을까

#### 1. 서비스에서 모든 api를 구현하는 케이스

1. admin client는 service에 구현된 토글 api를 이용하여 어떤 feature를 사용할 것인지 요청합니다.
2. 토글 api는 service(process)의 로컬 캐시와 데이터베이스에 어떤 feature을 사용할 것인지 저장하는 역할을 수행합니다.
3. app client가 요청을 할 경우 service는 feature에 맞는 로직을 수행합니다.
4. 만약 서비스가 다운되었다면 다시 시작하면서 ApplicationContext 로드 후 Database에 질의하여 어떤 feature가 활성화되었는지 체크 후 로컬 캐시에 저장합니다.

#### 2. 별도의 admin 서비스가 존재하여 토글을 구현하는 케이스

1. admin service를 통해 현재 사용할 feature를 on합니다.
2. 메시지 큐로 이를 전파하여 각 app service들은 현재 어떤 토글이 켜져있는지 감지 후 각자의 로컬 캐시에 feature값을 저장합니다.
    * api로 동기적인 호출을 시도할 수 있으나 app service가 많아질수록 admin service에서의 수정, 관리가 발생하기 때문에 queue를 이용한 전파 방식이 효율적인듯 합니다.
3. app client가 요청할 경우 app service는 feature에 맞는 로직을 수행합니다.
4. app service가 다운되었다면 다시 로드하면서 admin service로부터 현재 어떤 feature가 활성화되었는지 체크 후 각자의 로컬 캐시에 값을 다시 저장합니다.
