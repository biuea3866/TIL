# 클로드 코드 메모리

## 메모리란
---
메모리는 디스크에 저장되어 **세션을 넘어 살아남는 텍스트 파일들**의 집합이다. 클로드 코드는 매 세션 시작 시 메모리를 자동으로 시스템 프롬프트에 합쳐서, 모델이 이전 대화의 약속·사용자 선호·프로젝트 결정을 기억하는 것처럼 동작하게 만든다.

휘발성인 `system` 파라미터와 달리, 메모리는 디스크에 영구 저장되고 사용자별·프로젝트별로 관리된다. 시스템 프롬프트가 어떻게 메모리를 흡수하는지는 [claude_execution.md](claude_execution.md#system-시스템-프롬프트와-메모리)에서 다룬다.

## 위치와 디렉토리 구조
---
메모리는 두 갈래로 나뉜다.

```
[사용자 메모리 — auto memory]
~/.claude/projects/<프로젝트-경로>/memory/
  ├── MEMORY.md                       ← 인덱스 (항상 시스템 프롬프트에 자동 로드)
  ├── feedback_polite_speech.md       ← 필요 시 모델이 Read로 가져감
  ├── project_til_revisit_plan.md
  ├── user_role.md
  └── reference_*.md

[프로젝트 규칙]
<프로젝트>/CLAUDE.md                  ← 프로젝트 규칙 (시스템 프롬프트에 로드)
~/.claude/CLAUDE.md                   ← 사용자 글로벌 규칙
```

| 경로 | 범위 | 영속성 | 누가 편집 |
|---|---|---|---|
| `~/.claude/projects/<path>/memory/` | 사용자별·프로젝트별 | 영구 | 모델(auto memory) 또는 사용자 |
| `<프로젝트>/CLAUDE.md` | 프로젝트 전체 (팀 공유) | 영구 | 사람 (git 커밋) |
| `~/.claude/CLAUDE.md` | 사용자 글로벌 | 영구 | 사람 |

## MEMORY.md: 인덱스의 역할
---
`MEMORY.md`는 짧은 한 줄 링크만 모아두고, **실제 메모 내용은 별도 파일에 저장한다**.

```markdown
- [Polite speech](feedback_polite_speech.md) — 대화 시 존댓말 사용, 반말 금지
- [TIL 글 재정비 33일 일정](project_til_revisit_plan.md) — 5/7~6/8 매일 2개씩...
- [User role](user_role.md) — 10년차 Kotlin 개발자, React는 처음
```

이 인덱스 구조에는 세 가지 이유가 있다.

| 이유 | 설명 |
|---|---|
| **컨텍스트 절약** | MEMORY.md는 매 세션 무조건 로드됨. 풀텍스트를 다 넣으면 토큰 폭발 |
| **lazy loading** | 200줄짜리 인덱스만 미리 보고, 관련성 있는 메모만 모델이 `Read`로 가져감 |
| **캐시 안정성** | prefix가 안정적이어야 prompt cache hit rate가 올라감 |

> **200줄 제한**: MEMORY.md는 200줄을 넘으면 잘린다. 따라서 인덱스 자체는 가능한 한 짧게 유지하고 내용은 외부화하는 게 규약이다.

## 메모리 타입 (4종)
---
CLAUDE.md의 auto memory 정책에 따라 메모는 4가지 타입으로 분류된다.

| 타입 | 용도 | 예시 |
|---|---|---|
| **user** | 사용자의 역할·전문성·선호 | "10년차 Kotlin 개발자, React는 처음" |
| **feedback** | "이렇게 하지 마라 / 이렇게 해라" 지침 | "통합 테스트는 mock 금지, 실DB 써야 함" |
| **project** | 진행 중인 작업·결정·일정 | "5/7~6/8 TIL 재정비, 매일 2개" |
| **reference** | 외부 시스템 포인터 | "버그는 Linear INGEST 프로젝트에서 추적" |

### 1. user 메모리

사용자가 누구인지, 어떤 배경·역할·선호를 가졌는지를 기록한다. 모델이 답변 톤·깊이·예시를 사용자에게 맞추는 데 쓴다.

```markdown
---
name: User role
description: 사용자의 직무·경력·기술 스택
type: user
---

10년차 백엔드 개발자. Kotlin/Spring Boot 주력. JPA·QueryDSL 전문가.
React는 입문 단계 — 프론트 설명 시 백엔드 비유로 풀어줄 것.
```

### 2. feedback 메모리

사용자가 모델에게 준 교정·선호 지침을 기록한다. 같은 지적을 반복하지 않게 하는 메모.

```markdown
---
name: Polite speech
description: 대화 시 존댓말 사용, 반말 금지
type: feedback
---

대화 시 존댓말을 사용한다.

**Why:** 사용자가 존댓말 톤을 명시적으로 요청함.
**How to apply:** 모든 사용자 응답에 적용. 코드 주석·문서는 제외.
```

`Why:` / `How to apply:` 구조로 적으면 모델이 엣지 케이스에서 응용할 수 있다.

### 3. project 메모리

지금 진행 중인 작업·결정·일정을 기록한다. 세션 사이에 끊기지 않게 컨텍스트를 잇는 메모.

```markdown
---
name: TIL 재정비 계획
description: 5/7~6/8 매일 2편씩 통합·재작성
type: project
---

5월 7일 ~ 6월 8일까지 TIL 문서 33편을 매일 2편씩 통합·재정비한다.

**Why:** 흩어진 노트를 토픽별로 묶어 검색성을 높이기 위함.
**How to apply:** 새 세션에서 "오늘 작업"을 물으면 진행 번호 다음 항목 안내.
```

### 4. reference 메모리

외부 시스템 어디에 무엇이 있는지를 기록한다. "Jira는 여기, 로그는 여기" 같은 포인터.

```markdown
---
name: 파이프라인 버그 트래커
description: 데이터 파이프라인 관련 버그는 Linear INGEST 프로젝트에서 추적
type: reference
---

데이터 파이프라인·ETL 관련 이슈는 Linear의 "INGEST" 프로젝트에서 관리된다.

**Why:** 다른 팀과 분리된 백로그 운영.
**How to apply:** 파이프라인 버그 언급 시 자동으로 해당 프로젝트 참조.
```

## frontmatter 형식
---
모든 메모 파일은 동일한 frontmatter 구조를 따른다.

```yaml
---
name: <메모의 짧은 제목>
description: <한 줄 설명 — MEMORY.md 인덱스에 들어감>
type: <user | feedback | project | reference>
---

<본문 — 모델이 Read로 가져가는 내용>

**Why:** <이 규칙·정보가 생긴 이유>
**How to apply:** <언제·어떻게 적용해야 하는지>
```

| 필드 | 필수 | 역할 |
|---|---|---|
| `name` | ✅ | MEMORY.md 인덱스 링크의 제목 |
| `description` | ✅ | MEMORY.md 인덱스의 한 줄 설명 |
| `type` | ✅ | 메모리 분류 (4종 중 하나) |

본문에 **`Why:`** 와 **`How to apply:`** 를 적는 게 권장 규약이다. 단순 사실보다 "왜·언제" 라는 맥락이 들어가야 모델이 엣지 케이스에서 응용할 수 있다.

## CLAUDE.md vs MEMORY.md
---
같은 영속 메모리지만 성격이 완전히 다르다.

| 항목 | CLAUDE.md | MEMORY.md |
|---|---|---|
| 누가 관리하나 | 사람이 직접 편집·git 커밋 | 모델이 자동 갱신 |
| 내용 성격 | 프로젝트의 객관적 규칙 (코딩 컨벤션, 금지 패턴) | 사용자의 주관적 컨텍스트 (선호, 진행 상황) |
| 팀과 공유? | ✅ Yes (git 트래킹) | ❌ No (사용자별 로컬) |
| 변경 빈도 | 낮음 | 높음 |
| 형식 | 자유 마크다운 | 인덱스 + 분리된 메모 파일 |
| 토큰 부담 | 전체가 시스템 프롬프트에 들어감 | 인덱스만 들어가고 본문은 lazy load |

CLAUDE.md는 "이 프로젝트의 헌법"이고, MEMORY.md는 "당신과 나만 아는 메모장"이다.

## 무엇을 어디에 적어야 하나
---
| 상황 | 어디에 적어야 하나 |
|---|---|
| "이 프로젝트는 Kotlin Spring Boot입니다" | **CLAUDE.md** (프로젝트 규칙, 팀 공유) |
| "LocalDateTime 대신 ZonedDateTime을 써라" | **CLAUDE.md** (코딩 컨벤션, 팀 공유) |
| "나는 존댓말을 선호합니다" | **MEMORY.md (feedback)** (사용자 개인) |
| "이번 PR만 이 규칙으로 진행해줘" | 그냥 채팅으로 (어차피 messages로 누적되어 세션 동안 유효) |
| "어제 이 결정 내렸으니 다음 세션에서도 알아둬" | **MEMORY.md (project)** |
| "다른 프로젝트에서도 적용되었으면" | `~/.claude/CLAUDE.md` 또는 user-scope 메모리 |
| "Linear 프로젝트 ID 같은 외부 시스템 정보" | **MEMORY.md (reference)** |

판단 기준:

```
- 팀 전체에 적용되어야 하나?       → CLAUDE.md
- 나에게만 적용되어야 하나?        → MEMORY.md
- 이 세션만 유효하면 되나?         → 그냥 채팅
- 객관적 규칙인가?                 → CLAUDE.md
- 진행 상황·결정의 기록인가?       → MEMORY.md (project)
- 모델 행동을 교정하는 지침인가?   → MEMORY.md (feedback)
```

## auto memory: 자동 저장 정책
---
사용자가 명시적으로 요청하지 않아도, 모델이 다음 조건에서 자동으로 메모를 저장한다.

### 자동 저장 트리거

| 트리거 | 어떤 메모로? |
|---|---|
| 사용자가 자기 역할·전문성·선호를 드러내는 발화 | **user** |
| 사용자가 "다시는 X하지 마", "X는 항상 Y로" 같은 교정 발화 | **feedback** |
| 모델이 제안한 접근을 사용자가 명시적으로 칭찬 (예: "그게 정답이야") | **feedback** (success pattern) |
| 일정·마감·할당 정보가 등장 | **project** |
| 외부 시스템 위치·식별자 정보가 등장 | **reference** |

### 자동 저장 흐름

```
사용자 발화
   │
   ▼
모델이 "이건 저장할 만한가" 판정
   │
   ├─ Yes → 메모리 파일 생성 또는 기존 파일 업데이트
   │         └─ MEMORY.md 인덱스에 한 줄 추가/수정
   │
   └─ No  → 그냥 응답만
```

### 자동 저장하지 말아야 할 것

- 코드 패턴·아키텍처·컨벤션 → CLAUDE.md 또는 코드를 직접 읽으면 알 수 있음
- git 이력에서 알 수 있는 변경 정보
- 디버깅 솔루션 — 코드와 커밋 메시지에 이미 있음
- 일시적 작업 상태 — TodoWrite로 처리

## 메모리 라이프사이클
---
한 메모의 생애주기는 다음과 같다.

```
[발화]
사용자: "통합 테스트에서 mock 쓰지 마. 작년에 그것 때문에 prod 사고 났어."
   │
   ▼
[저장]
모델: feedback_no_mock_integration.md 생성
      MEMORY.md에 "- [No mock in integration tests](...) — 사고 재발 방지" 추가
   │
   ▼
[세션 종료]
   디스크에 남음
   │
   ▼
[다음 세션 시작]
   Context Manager가 MEMORY.md 자동 로드
   → system 프롬프트에 인덱스 포함
   → 모델이 "통합 테스트 짜줘" 요청 받으면 자동으로 mock 회피
   │
   ▼
[갱신 또는 폐기]
   - 규칙이 진화하면 → 모델이 해당 파일 업데이트
   - 더 이상 유효하지 않으면 → 사용자가 "그 메모 잊어줘" 요청 → 파일 삭제 + 인덱스 정리
```

## 운영 팁
---

| 팁 | 이유 |
|---|---|
| **한 사람당 1세션 권장** | 같은 사용자가 여러 세션을 동시에 열면 MEMORY.md 갱신이 경쟁할 수 있음 |
| **MEMORY.md를 가끔 직접 열어보기** | 모델이 자동 저장한 메모가 의도와 맞는지 확인 |
| **잘못된 메모는 명시적으로 폐기** | "그 메모 잊어줘" 라고 요청하면 모델이 파일 삭제 + 인덱스 정리 |
| **공유 규칙은 무조건 CLAUDE.md로** | MEMORY.md에 적으면 다른 팀원은 못 봄 |
| **시간 정보는 절대 날짜로** | "내일", "지난주" 대신 "2026-05-18" 처럼 적어야 시간이 지나도 의미 보존 |

## 한 줄 요약
---
> **MEMORY.md는 사용자별로 디스크에 남는 영속 메모의 인덱스이며, Context Manager가 매 세션 자동으로 시스템 프롬프트에 합쳐서 모델이 과거를 기억하는 것처럼 동작하게 만든다. CLAUDE.md가 프로젝트의 객관적 헌법이라면, MEMORY.md는 사용자와 모델 사이의 사적 메모장이다.**
