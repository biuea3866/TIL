## AI 오케스트레이션이란

---

### AI 오케스트레이션

AI 오케스트레이션이란 여러 AI 모델, 애플리케이션, 데이터 소스, 에이전트들을 워크플로 안에서 유기적으로 연결, 조정 관리하는 기술이다. 여기서의 워크플로는 파이프라인이며, 단계별로 작업을 수행한다. 여기서의 작업은 순차적으로 수행할수도, 병렬적으로 수행할수도 있다. 어떤 워크플로를 구성하느냐에 따라 선택할 수 있는 옵션은 다양하다.

초기 AI 서비스가 등장했을 때 사용자는 보통 하나의 질문과 하나의 답변을 받고, 이 과정이 연쇄적이고 반복적으로 수행이 되었다. 이 단순한 흐름 자체만으로 유의미한 결과를 낼 순 있으나 더욱 복합적으로 에이전트들을 동시에 활용하기 위해 AI 오케스트레이션이 등장하게 되었다.

### AI 오케스트레이션 프레임워크

#### CrewAI

crewAI는 여러 개의 AI 모델을 하나의 팀처럼 구성하고, 협업하게 해주는 프레임워크이다. 에이전트끼리 대화, 작업 위임, 결과 검토 등 프로세스를 유기적으로 진행할 수 있다는 장점이 존재한다.

**1. 에이전트**

팀의 구성원이다. 각 에이전트는 역할(Role), 목표(Goal), 배경(Backstory)을 가진다.

예시: 너는 10년차 IT 전문 기자야. 기술 트렌드를 분석하고 기사를 쓰는 게 목표야.

**2. 태스크**

에이전트가 수행할 업무이다. 업무 설명과 기대 결과물을 정의하고, 특정 에이전트에게 할당한다.

**3. 도구**

에이전트가 사용하는 도구이다. 검색 엔진, 데이터베이스 조회, 파일 입출력 등 AI가 실제 수행할 수 있는 액션이다.

**4. 크루**

모든 것을 조율하는 팀이다. 어떤 에이전트와 태스크를 합칠지 작업을 순차적으로 할지 등을 결정한다.

가령 일하는 방식을 예로 들면 다음과 같다.

단순히 AI에게 AI 트렌드 블로그 써줘라고 하면 에이전트 혼자서 작업을 하지만 CrewAI는 다음과 같이 작업을 수행한다.

- 에이전트 A (리서처): 최신 AI 뉴스 5개를 구글에서 검색하여 핵심 내용을 요약한다.

- 에이전트 B (필자)**:** 에이전트 A의 요약본을 받아 독자가 읽기 쉬운 블로그 글로 작성한다.

- 에이전트 C (편집자)**:** 필자가 쓴 글의 오타를 확인하고, 내용이 정확한지 검수한 뒤 최종 승인한다.

이 과정을 오케스트레이션이라 하며, 에이전트끼리 이 내용이 부족하니 다시 조사해와라고 피드백을 주고받기도 한다.

#### LangChain

LangChain은 AI 모델을 다양한 데이터 소스나 도구들을 연결하여 작업을 수행한다. 하나의 에이전트로 질문과 답변을 수행할 경우 정보의 부재가 있을 수 있다. 가령 특정 날짜 이후 데이터가 자체 모델에 학습되지 않았을 경우, 사내 데이터에 대한 정보가 없을 경우 등 다양한 경우가 존재하는데 이 한계를 보완해주기 위해 파이프라인을 구성해주는 것이 핵심이다.

**1. Model I/O**

다양한 LLM(OpenAPI, Anthropic, HuggingFace 등)을 일관된 방식으로 호출하고 관리한다.

**2. Retrieval (RAG)**

외부 데이터(PDF, DB, 웹 페이지)를 가져와 AI가 참고할 수 있게 변환하고 검색한다.

**3. Chains**

여러 구성요소를 결합한다. (데이터 검색 -> 요약 -> 번역 등을 하나의 워크플로로 구성)

**4. Agents**

AI가 스스로 어떤 도구를 사용할지 판단하고 실행 순서를 결정하게 한다.

**5. Memory**

대화의 맥락을 기억하여 이전 질문과 답변을 다음 단계에 활용한다.

**6. Callbacks**

작업의 중간 단계를 로깅하거나 모니터링한다.

#### Claude Agent Team

claude code에서 제공하는 기능이다. 이미 subagent라는 기능이 있었고, 이들은 각자 병렬적으로 일할 수 있었다. 하지만 병렬적으로 작업하기만 하고, 상위 메인 에이전트에게 결과를 보고하기만 하고 종료하는 방식이다. 즉, subagent끼리의 협업과 소통은 불가하며 main agent가 모든 것을 통제해야한다. Agent Team은 하위 팀원들이 서로 의견을 나눌 수 있고, 작업을 공유할 수 있다. 이 점에서 subagent와 큰 차이를 보인다. 위의 프레임워크는 아니지만 claude에서 제공하는 자체 기능이므로 병렬적으로 일을 할 수 있다는 점에서 ai orchestration과 같다고 생각한다.

**1. 팀리더**

팀을 만들고, 팀원들을 생상하며 작업을 조율하는 메인 세션이다.

**2. 팀원들**

할당된 작업들을 각각 동작하는 별도 인스턴스이다.

**3. 작업 목록**

팀원들이 요청하고, 완료하는 공유 작업 항목 목록이다.

**4. 메일박스**

에이전트간 통신을 위한 메시징 시스템이다.

### 프로세스 정의

![](./img/ai_1.png)

AI 오케스트레이션을 적용하기 전에 우선 어떤 프로세스로 작업을 수행할지를 정의하였다. 처음 러프하게 정의한 프로세스는 다음과 같다.

PRD 요구사항 분석 -> 필요 기술 스택 분석 -> 구현 티켓 나누기 -> 티켓별 테스트 케이스 작성 -> 코드 생성 -> 코드 리뷰

러프하게 프로세스를 정의하고, 외에 필요한 프로세스 보충을 한 결과는 위의 이미지와 같다. 이를 기반으로 여러 에이전트가 병렬적으로 작업을 수행할 수 있는 Claude Agent Team를 활용하기로 한다.

* 문서 관리 도구: Jira를 이용하여 티켓 단위로 업무를 수행한다. Confluence에서는 문서들을 관리한다.

* 에이전트 병렬 처리 도구: worktree를 이용하여 각 에이전트들이 git에 형상관리를 한다.

* 역할: pm, fe, be, devops들의 역할을 에이전트들이 번갈아가며, 수행한다.

* 작업량: 하루에 진행 가능한 할당량을 두고, 작업 수행 (e.g. be 티켓 1개, devops 티켓 1개 등)

플로우를 정의하고, 어떤 역할들이 있고, 어떤 도구들을 활용할지에 대한 정의가 필요하다. 이 정의까지 완료되어야 에이전트들이 역할을 부여받고, 어떤 도구로 일을 할지 상세 지침으로 알 수 있기 때문이다.

## 후기

---

처음부터 상상한 것 그대로 만들기는 역시 어려운 부분이 많았다. 때문에 중간중간 ai에게 요구사항을 반영시키고, 업데이트하는 과정이 많았다.

### 작업 완료의 기준은?

보통 업무에서 하나의 작업은 코드를 작성하고, pull request를 올리고, 머지되고 까지 어떤 플로우가 존재한다. 이 작업 완료의 기준을 정의해주지 않으면, 작업을 중간중간 멈추면서 다음 작업을 진행할까요? 라는 질문을 계속 던졌다. 이를 방지하기 위해 작업 완료의 기준을 잡고, 이를 따르게 하였다.

### 왜 처음 요구한대로 안하지?

가장 처음에 요구한 것은 당연하게도 AI Wiki 프로젝트를 작업하는 것이었다. 하지만 ai orchestration을 기반으로 작업할 것이라고 하였더니 어느샌가 AI가 자신이 작업할 프로젝트가 workflow라고 이해를 해버리는 일이 발생했다. 자신이 작업할 프로젝트가 아예 다른 것이니 이상한 작업을 진행했고, 다시 바로잡는 질의를 수행하였다.

결과적으로 정의한 파이프라인에 PRD를 읽도록 고정이 되었고, 이를 기반으로 작업하게 되었다.

### 왜 정의한 파이프라인을 안따라가는 것이지?

파이프라인을 전부 정의하고, claude에게 지시한 사항은 팀장1명과 팀원3명을 구성하여 위키 프로젝트 작업을 수행하란느 지시였다. 하지만 첫 작업이 시작되면서 팀원들이 파이프라인을 따르지 않고, 각자가 작업을 수행하기 시작했다. 이를 막기 위해 파이프라인을 따르도록 고지했고, 팀원들의 모든 작업을 멈춘 뒤 파이프라인대로 작업이 수행되었다.

아마 CLAUDE.md 파일에 파이프라인을 따르라는 것이 명시되어있지 않아 생긴 문제인듯하다.

---

요구사항을 계속해서 업데이트함에 따라 프로세스 스텝이 업데이트되고, 문서와 티켓을 작성하는 방법들이 점점 고도화되는 것이 눈에 띄게 보이게 되었다. 뭐든 그렇듯 처음에는 의도한대로 동작하지 않는 것이 많지만 점점 나아지는 모습들을 볼 수 있다. 이번 wiki 프로젝트를 통해 느낀 것은 만들 제품이 무엇인지 정의하고, 제품을 만들기 위한 파이프라인이 정의된다면 110달러(claude max 5배짜리 플랜)만으로 0 -> 1로 가는 과정이 굉장히 빨라졌다는 것이다.

그리고 추가적으로 로드맵을 구상하고, 로드맵 문서에 녹이면 이걸 파이프라인으로 다시 소화할 수 있다는 점에서 지속적인 제품 디벨롭도 가능하다는 점이 가장 인상깊었다.

이번엔 가볍게 AI 오케스트레이션 기술(Claude Agent Team)을 이용해서 제품을 만들어 보았다. 파이프라인을 구성하고, 역할을 정의하고, 목표를 구성함으로써 제품을 만들 수 있었는데 비단 제품을 만들 때만 사용 가능하다는 생각은 들지 않는다. 문제, 역할, 파이프라인이라는 추상에 어떤 구현을 넣느냐에 따라서 다양하게 활용이 가능해 보인다.
