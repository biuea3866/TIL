# 클로드 코드의 토큰
## 토큰이란
---

토큰은 LLM이 텍스트를 처리하는 최소 단위다. 모델은 문장을 그대로 읽는 것이 아니라, 토크나이저로 텍스트를 잘게 쪼갠 토큰 시퀀스로 변환한 뒤 처리한다.

- 영어 기준 대략 `1토큰 ≈ 4글자 ≈ 0.75단어`
- 한국어, 코드, 특수문자는 토큰 효율이 낮아 같은 의미라도 토큰을 더 많이 소비한다.
- 예: `"Hello"`는 1토큰이지만, `"안녕하세요"`는 보통 3~5토큰으로 쪼개진다.

클로드 코드는 Claude API를 호출하는 클라이언트이고, API 호출 비용·속도·컨텍스트 한계가 모두 이 토큰 수로 결정된다. 

## 토큰의 종류
---

클로드 코드가 한 번 API를 호출할 때 토큰은 크게 두 방향으로 흐른다.

|종류|무엇인가|특징|
|-|-|-|
|Input(입력) 토큰|모델에게 보내는 모든 것|시스템 프롬프트, 도구 정의, 대화 기록, 파일 내용, 도구 결과 전부 포함|
|Output(출력) 토큰|모델이 생성하는 것|텍스트 응답, 도구 호출(tool_use), thinking 블록|

여기에 프롬프트 캐싱(prompt caching)이 더해지면서 입력 토큰이 다시 두 갈래로 나뉜다.

|종류|무엇인가|상대 비용|
|-|-|-|
|Cache write|캐시에 새로 기록하는 입력 토큰|일반 입력보다 비쌈 (약 1.25x)|
|Cache read캐시에서 읽어오는 입력 토큰|일반 입력보다 훨씬 쌈 (약 0.1x)|

> **프롬프트 캐싱**
모델에게 프롬프트를 보낼 때 크게 변하지 않는 (CLAUDE.md, 도구 스키마, Hook, Memory 등)을 캐싱하여 사용


## 클로드 코드는 토큰을 어떻게 쓰는가
---

### 1. 매 턴, 컨텍스트를 새로 조립한다

LLM은 상태가 없다(stateless). 이전 대화를 모델이 "기억"하는 것이 아니라, 클로드 코드가 매 요청마다 누적된 전체 컨텍스트를 다시 통째로 입력으로 보낸다. 한 번의 호출에 들어가는 입력 토큰은 다음과 같다.

```
[ 입력 토큰 ]
 ├─ 시스템 프롬프트      (클로드 코드 정체성 · 하네스 규칙)
 ├─ 도구 스키마          (Bash, Read, Edit ... + 연결된 모든 MCP 도구 정의)
 ├─ CLAUDE.md / 메모리   (프로젝트 지침 · 자동 메모리)
 ├─ 대화 기록            (지금까지의 모든 user/assistant 메시지)
 ├─ 도구 결과            (Read한 파일 내용, Bash 출력, Grep 결과 ...)
 └─ 이번 사용자 입력
```

### 2. 대화가 길어질수록 입력이 선형으로 늘어난다

매 턴마다 전체를 다시 보내므로, N번째 턴의 입력은 `1턴 ~ (N-1)턴의 모든 내용 + 새 입력`이다.

```
턴 1:  [고정 prefix] + 입력1                         → 적음
턴 2:  [고정 prefix] + 입력1 + 응답1 + 입력2          → 누적
턴 3:  [고정 prefix] + 입력1 + 응답1 + 입력2 + 응답2 + 입력3
...
턴 N:  앞의 모든 것이 그대로 다시 입력으로 들어감     → 점점 비쌈
```

즉, 같은 대화 안에서 뒤로 갈수록 한 번의 응답이 점점 비싸진다. 이 때문에 세션이 길어질수록 토큰 사용이 커진다.

### 3. 프롬프트 캐싱이 비용을 눌러준다

컨텍스트의 앞부분(시스템 프롬프트, 도구 스키마, CLAUDE.md 등)은 턴마다 거의 바뀌지 않는다. 클로드 코드는 이 prefix를 캐싱해서, 다음 턴에는 그 부분을 `cache read`(약 0.1x)로 저렴하게 재사용한다.

- 캐시는 5분 TTL을 가진다. 5분 안에 다음 요청이 오면 캐시 적중, 넘기면 캐시가 만료되어 다시 `cache write`로 비싸게 재기록된다.
- 그래서 컨텍스트 앞부분을 흔드는 행위(중간에 도구를 추가/제거, CLAUDE.md 수정 등)는 캐시를 사용하지 못하기에 비용을 키운다.

## 토큰을 많이 쓰는 상황
---

|상황|왜 비싼가|
|-|-|
|**대용량 파일 전체 Read**|수천 줄 파일을 통째로 읽으면 그 내용이 입력에 적재되고, 이후 모든 턴에서 계속 재전송된다|
|**긴 세션 누적**|위 2번 원리. 대화가 길수록 매 턴 입력이 커진다|
|**많은 MCP 서버 연결**|MCP 도구 스키마는 매 요청마다 입력에 포함된다. 도구가 수십 개면 그 정의만으로 수천~수만 토큰을 고정 소모|
|**넓은 검색 결과 적재**|`grep -r`로 수백 개 매치, 큰 디렉토리 `ls -R` 등 결과 전체가 컨텍스트로 들어감|
|**빌드/테스트 로그 통째 적재**|실패 로그 전체를 그대로 받으면 수만 토큰. 대부분은 노이즈|
|**시행착오 반복**|실패한 명령·잘못된 출력이 대화 기록에 그대로 쌓여 이후 턴마다 재전송된다|
|**이미지·PDF**|이미지는 해상도에 비례해 토큰화되고, PDF 다중 페이지는 페이지마다 누적된다|
|**장황한 출력 요구**|출력 토큰은 단가가 비싸다. 불필요하게 긴 응답/전체 코드 재출력은 비효율적|

## 구체적인 상황
---

### 상황 1. 코드 작업을 할 때

"`OrderService`의 결제 취소 버그를 고쳐줘" 라는 요청 하나를 처리하는 흐름이다.

```
[고정 prefix]  시스템 프롬프트 + 도구 스키마 + CLAUDE.md  ≈ 12,000 토큰  (캐시되어 2턴부터 cache read)

턴 1  사용자: "결제 취소 버그 고쳐줘"
      → Read OrderService.kt (600줄)        +  8,000  (입력에 적재)
      → Read PaymentClient.kt (300줄)        +  4,000
      → Grep "cancel" 결과 40개 매치          +  2,000

턴 2  →  /test 실행, 실패 스택트레이스 전체   + 15,000  (로그 노이즈가 큼)

턴 3  → Edit OrderService.kt                  +  1,500  (출력: 수정 diff)
      → /test 재실행, 또 실패                 + 15,000

턴 4  → Edit 다시                             +  1,500
      → /test 통과                            +  3,000
```

여기서 턴 4 시점의 입력 = 앞의 모든 것이 다시 들어간다.

```
턴 4 입력 ≈ 고정 prefix(캐시) + 파일 14,000 + 실패로그 30,000 + 이전 응답들 + 새 입력
         → 한 번의 응답을 위해 누적 6만 토큰 이상을 매번 재전송
```

- 600줄짜리 파일을 통째로 Read → 정작 버그는 20줄인데 580줄이 모든 턴에 따라다닌다.
- 실패 스택트레이스 전체를 두 번 받음 → 대부분 동일한 프레임워크 노이즈인데 3만 토큰이 누적.
- 시행착오(Edit→실패→Edit)가 대화 기록에 쌓여 이후 턴마다 재전송.

**개선안**
- `Grep`으로 `cancel` 위치를 먼저 찾고 → 해당 함수 구간만 `Read(offset/limit)`.
- 테스트는 `./gradlew test --tests OrderServiceTest 2>&1 | tail -40` 처럼 실패 로직만 컨텍스트에 올린다.
- 버그 원인을 한 번에 짚어 Edit 횟수를 줄인다(시행착오 = 누적 비용).
- 버그 수정이 끝나면 다음 작업 전에 `/clear`.

### 상황 2. MCP 서버를 연결해서 분석할 때

DB MCP, GitHub MCP, Atlassian(Jira/Confluence) MCP 등을 붙여 "이 데이터/이슈를 분석해줘"를 하는 흐름이다.
```
[연결 즉시 발생하는 고정 비용]
  MCP 서버 A 도구 20개 스키마
  MCP 서버 B 도구 35개 스키마        → 도구 정의만으로 ≈ 15,000~30,000 토큰
  MCP 서버 C 도구 15개 스키마           (한 줄도 호출하기 전에 매 요청 고정 소모)

턴 1  사용자: "지난 분기 주문 테이블 분석해줘"
      → MCP query: SELECT * ... LIMIT 1000   + 40,000  (행 전체가 결과로 적재)

턴 2  → MCP: Jira 이슈 50개 조회             + 25,000  (이슈 본문·댓글까지 통째로)

턴 3  → 분석 응답 생성                        +  4,000  (출력)
```

1. 도구 스키마 고정 비용: MCP는 연결만 해도 모든 도구 정의가 매 요청 입력에 들어간다. 서버 3개 = 도구 70개면, 질문을 하기도 전에 매 턴 2~3만 토큰을 깔고 시작한다.
2. 결과 셋이 통째로 적재: `SELECT *`나 "이슈 전부 가져오기"는 수백 행/수십 문서를 그대로 컨텍스트에 올린다. 대부분은 분석에 쓰지도 않는 열·필드다.
3. 인증/세션이 헤드리스에서 빠질 수 있음: 일부 MCP는 대화형 인증이 필요해 cron·백그라운드 실행에서 누락되며, 재시도가 또 토큰을 쓴다.

**개선안**
- 쓸 MCP만 연결: 이번 작업에 Jira가 필요 없으면 끈다 → 도구 스키마 고정 비용이 통째로 사라진다.
- 쿼리에서 필요한 열·`LIMIT`을 명시: `SELECT *` 대신 `SELECT id, status, amount ... LIMIT 50`.
- 대량 분석은 서브에이전트에 위임: raw 결과는 서브 컨텍스트에서 소화하고, 메인에는 집계·결론만 반환받는다.
- 클로드 코드의 tool search(지연 로드)활용: 도구 스키마를 항상 깔지 않고 필요할 때만 불러와 고정 비용을 낮춘다.

## 절약 방법
---

### 1. 탐색은 서브에이전트로 격리한다
넓은 코드 탐색·검색은 `Agent`(Task)나 `Explore` 서브에이전트에 맡긴다. 서브에이전트는 자기만의 컨텍스트에서 수십 개 파일을 뒤지고 결론만 메인 대화로 반환한다. 메인 컨텍스트에는 파일 덤프가 아니라 요약만 남으므로, 이후 모든 턴의 재전송 비용이 줄어든다.

### 2. 파일은 필요한 부분만 읽는다
`Read`에 `offset`/`limit`을 주거나, `Grep`으로 위치를 먼저 좁힌 뒤 해당 구간만 읽는다. 전체 Read는 큰 낭비다.

### 3. 대화를 적극적으로 비우거나 압축한다
- `/clear` : 대화 기록을 초기화해 누적 입력을 리셋한다. 작업이 바뀌면 가장 확실한 절약.
- `/compact` : 맥락을 유지한 채 대화를 요약 압축해 토큰을 줄인다.
- 한 세션에서 무관한 작업을 길게 이어가지 말고, 작업 단위로 끊는다.

### 4. 쓰지 않는 MCP 서버는 끈다
도구 스키마는 매 요청 고정 비용이다. 안 쓰는 MCP 서버를 비활성화하면 모든 턴에서 그만큼 입력이 줄어든다. (클로드 코드의 deferred tool / tool search는 도구 정의를 지연 로드해 이 비용을 더 낮춘다.)

### 5. 검색은 좁혀서 한다
`grep -r` 전체 스캔 대신 경로·확장자·패턴으로 범위를 좁힌다. 로그는 `| tail`, `| head`, `grep`으로 핵심만 컨텍스트에 올린다.

### 6. 캐시를 살린다
- 연속 작업은 5분 안에 요청하여 캐시 적중률을 높인다.
- 세션 중간에 CLAUDE.md/도구 구성을 자주 바꿔 prefix를 흔들지 않는다.

### 7. 한 번에 정확히 요청한다
모호한 요청 → 시행착오 → 실패 출력 누적이 가장 비싼 패턴이다. 요구사항을 구체적으로 한 번에 전달할수록 재시도로 쌓이는 토큰이 줄어든다.

### 8. 출력도 비용임을 기억한다
전체 파일을 다시 출력보다 diff/부분 수정을 요청한다. 출력 토큰 단가가 입력보다 비싸므로 장황한 응답은 직접적인 낭비다.
