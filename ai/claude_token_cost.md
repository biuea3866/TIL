# 클로드 코드 토큰 비용 모델

## 왜 비용 모델을 이해해야 하나
---
같은 작업이라도 **세션을 어떻게 운영하느냐에 따라 토큰 비용이 25배까지 차이 난다**. 베스트 케이스는 호출당 0.1x, 워스트 케이스는 2.5x + 추가 요약 비용까지 붙는다.

이 격차는 세 가지 임계치에 의해 만들어진다.

- **프롬프트 캐시 TTL (5분)** — TTL 안에 재호출하면 prefix 단가 90% 할인
- **Long-context 가격 임계치 (200K)** — 200K 토큰 초과 시 단가 2배
- **자동 압축 임계치 (~95%)** — 컨텍스트가 차면 LLM 요약 실행 + 캐시 무효화

세션을 짧게·자주·연속적으로 쓰면 캐시가 살아남고, 길게·드물게·산발적으로 쓰면 캐시 미스와 압축이 누적된다. 클로드 코드의 컨텍스트 관리 동작 원리 자체는 [claude_execution.md](claude_execution.md#컨텍스트-관리)에서 다룬다.

## 프롬프트 캐시 가격 구조
---
Anthropic의 프롬프트 캐시 가격은 다음과 같다 (input 기준 상대값).

| 항목 | 단가 (일반 input = 1x) |
|---|---|
| 일반 input | 1x |
| **Cache write** (캐시에 처음 적재, 5분 TTL) | 1.25x |
| **Cache read** (캐시 hit) | **0.1x (90% 할인)** |
| 1시간 TTL cache write | 2x |
| 1시간 TTL cache read | 0.1x (동일) |
| output | 별도 (모델·길이 따라) |

### 캐시 동작 흐름

```
[첫 호출 = cache miss]
  system + tools 전체를 cache write 단가(1.25x)로 보냄
  → 캐시에 저장됨, TTL 5분 시작

[5분 안 재호출 = cache hit]
  system + tools 부분은 cache read 단가(0.1x)로 처리
  → 입력 토큰의 90% 할인

[5분 지난 후 재호출 = cache miss]
  처음부터 다시 cache write → 또 1.25x 비용
```

5분 TTL이 비용에 가장 큰 영향을 준다. 시스템 프롬프트가 수만 토큰인 클로드 코드 환경에서는, **캐시 hit 여부 하나로 호출당 비용이 10배 이상 차이**난다.

## 세 가지 임계치
---

| 임계치 | 기준 | 효과 |
|---|---|---|
| **자동 압축 임계치** | 컨텍스트 윈도우의 ~92-95% | CLI가 prior turns를 LLM 요약으로 압축 → 추가 토큰 소모 |
| **Long-context 가격 임계치** | 입력 토큰 **200K** 초과 (1M 모델 한정) | 해당 호출의 input/output 단가가 **2배** |
| **프롬프트 캐시 TTL** | 마지막 호출로부터 **5분** | TTL 안: prefix 단가 0.1x / TTL 밖: 매번 다시 풀가격 |

### 1) 자동 압축 임계치 (~95%)

```
컨텍스트 사용량
     │
100% ┼─────────────────────── [컨텍스트 오버플로우]
     │
 95% ┼─────────╳─────────── ← 자동 압축 트리거
     │       ╱│
     │     ╱  │  요약 LLM 호출 (수만 토큰 소모)
     │   ╱    ▼
 30% ┼─╳      ⤓
     │ │   압축 후 메시지 배열 축소
     │ │
  0% ┼─┴──────────────────────────► 시간
```

압축 시 소모되는 토큰:
- 압축 자체가 별도 LLM 호출 → 보통 수만 토큰의 input + 수천 토큰의 output 추가 소모
- 압축 후 prefix가 바뀌므로 **프롬프트 캐시 hit이 깨짐** → 다음 호출에서 시스템 프롬프트·도구 스키마까지 다시 풀가격으로 보냄
- 즉 "한 번의 압축" = "요약 비용 + 다음 호출의 캐시 미스 비용" 이중 부담

Claude Code UI 표시: `[Compaction occurred at HH:MM]` 가 보이면 이 순간이 일어난 것.

### 2) Long-context 가격 임계치 (200K)

1M 컨텍스트 모델(`claude-opus-4-7[1m]`, `claude-sonnet-4-6[1m]`)에서 **입력이 200K를 넘으면 그 호출의 단가가 2배**가 된다.

```
입력 토큰 단가 (1M 컨텍스트 모델 기준)
     │
  2x ┤              ┌──────────────
     │              │
  1x ┤──────────────┘
     │
     └──────────────┴──────────────► 입력 토큰
                  200K           1M
```

| 입력 크기 | input 단가 | output 단가 |
|---|---|---|
| ≤ 200K (전체 1M의 20%) | 1x (기본) | 1x (기본) |
| > 200K | **2x** | **2x** |

요점: **199K → 201K로 넘어가는 순간 비용이 두 배가 된다.** 컨텍스트의 절반(500K)을 쓰는 게 5x가 아니라 사실상 10x인 셈이다.

### 3) 프롬프트 캐시 TTL (5분)

위의 [프롬프트 캐시 가격 구조](#프롬프트-캐시-가격-구조) 참고.

이래서 시스템 프롬프트에 **"5분 안 자고 5분 넘어가면 폭망"** 룰이 존재한다. 사용자가 30분 자리를 비웠다가 돌아와 한 번 호출하면, 그 한 번의 호출에 거의 풀가격의 prefix가 다시 들어간다.

## 결합 효과: 25배 격차
---
가장 위험한 시나리오는 **세 임계치가 한꺼번에 터질 때**다.

```
시나리오: 큰 코드베이스에서 1시간 작업

t=0     세션 시작, 컨텍스트 5%
t=20    파일 여럿 읽음, 컨텍스트 40%
t=40    더 읽음, 컨텍스트 75%
t=45    자동 압축 트리거 (~95%)
        → 요약 LLM 호출 (+수만 토큰)
        → 캐시 prefix 깨짐
t=46    다음 호출
        → cache miss + 새 prefix 전체 풀가격
        → 만약 압축 후에도 입력이 200K 넘었으면 → 2x 단가 적용
        → 트리플 펀치
```

세 임계치를 누적 효과로 보면:

| 상황 | 한 호출 비용 (기본 대비) |
|---|---|
| 캐시 hit + 200K 이하 | ~0.1x (이상적) |
| 캐시 miss + 200K 이하 | ~1.25x |
| 캐시 hit + 200K 초과 | ~0.2x (절반은 캐시되어도 2배 단가) |
| 캐시 miss + 200K 초과 | **~2.5x** |
| 압축 직후 + 200K 초과 | **~2.5x + 요약 비용** |

베스트 케이스와 워스트 케이스 사이의 **격차가 25배** 정도 된다.

## 실전 권장 패턴
---
이 비용 구조에서 살아남으려면 다음 전략을 의식해야 한다.

| 전략 | 이유 |
|---|---|
| **5분 안에 후속 질문하기** | 캐시 hit으로 90% 할인 유지 |
| **5분 넘기면 차라리 30분+ 자기** | 한 번의 캐시 미스로 끝, 자잘한 미스 누적 회피 |
| **컨텍스트 70% 전에 정리** | 자동 압축 트리거를 피하면 캐시도 살아남음 |
| **큰 파일 탐색은 서브에이전트로** | 메인 컨텍스트에 누적되지 않아 200K·압축 임계치 도달 지연 |
| **TodoWrite로 진행 상태 외부화** | 압축이 일어나도 작업 맥락 손실 최소화 |
| **200K 직전에는 새 세션 시작** | 2x 단가 회피 + 압축 비용 회피 |

### 짧고 자주 vs 길게 한 번

```
[패턴 A: 짧고 자주 (5분 내 연속)]
호출 1: cache miss (1.25x)
호출 2: cache hit  (0.1x)
호출 3: cache hit  (0.1x)
호출 4: cache hit  (0.1x)
호출 5: cache hit  (0.1x)
────────────────────────────
평균:   ~0.33x

[패턴 B: 길게 띄엄띄엄 (10분마다)]
호출 1: cache miss (1.25x)
[10분 대기]
호출 2: cache miss (1.25x)
[10분 대기]
호출 3: cache miss (1.25x)
────────────────────────────
평균:   1.25x  → 패턴 A의 ~4배
```

### 컨텍스트 사용량 관리

```
[안전 영역: 0~70%]
  - 압축 위험 없음
  - 캐시 prefix 안정적
  → 자유롭게 작업

[주의 영역: 70~85%]
  - 곧 압축 임박
  - 서브에이전트로 작업 위임 시작
  → 큰 파일은 Agent("Explore", ...)에 맡기기

[위험 영역: 85~95%]
  - 압축 트리거 직전
  - TodoWrite로 진행 상태 외부화
  → 작업 완료 후 새 세션 시작 권장

[압축 발생: ~95%]
  - 요약 비용 + 캐시 미스 = 트리플 펀치
  → 가능하면 그 전에 끊기
```

### 세션 끊고 이어가기

자동 압축이 반복되어도 한계가 온다. 압축은 무한 반복할 수 없고, 누적될수록 "요약 위에 또 요약"이 되어 품질이 급격히 떨어진다. 어느 시점에 클로드 코드가 **"컨텍스트가 부족합니다, 새 세션에서 작업하세요"** 안내를 띄운다.

```
세션 시작 → 컨텍스트 95% → 1차 압축 → 다시 차오름
                          → 2차 압축 → 더 잘 안 줄어듦
                          → 3차 압축 → 거의 효과 없음
                          → 🛑 "새 세션에서 작업하세요"
```

핵심: **압축은 무한 반복할 수 없다.** 이미 요약된 메시지는 더 줄일 게 없고, 줄여봤자 의미가 흐려져 작업을 계속할 수 없는 상태가 된다.

#### 새 세션에서 사라지는 것 / 유지되는 것

| 항목 | 새 세션에서? |
|---|---|
| 대화 이력 (messages 배열) | ❌ 사라짐 |
| TodoWrite 진행 상태 | ❌ 사라짐 |
| 모델이 발견한 파일 위치·이해 | ❌ 사라짐 |
| 디버깅 가설·중간 추론 | ❌ 사라짐 |
| 코드 변경 사항 (디스크) | ✅ 유지 |
| git 히스토리·브랜치 상태 | ✅ 유지 |
| CLAUDE.md (프로젝트 규칙) | ✅ 자동 로드 |
| MEMORY.md (사용자 메모) | ✅ 자동 로드 |
| 환경 정보 (cwd, git status) | ✅ 자동 스캔 |

핵심: **코드와 메모리는 디스크에 남아 있다.** 사라지는 건 "이 세션에서만 임시로 유지되던 상태"뿐이다.

#### 매끄럽게 넘어가는 3단계

```
1단계: 현재 세션 종료 직전
  "지금까지 진행한 작업을 MEMORY.md project 메모로 저장해줘"
  → 진행 상황·다음 할 일·핵심 결정이 디스크에 저장됨

2단계: 새 세션 시작
  SessionStart 훅이 이전 세션 요약 자동 주입
  Context Manager가 MEMORY.md 자동 로드

3단계: 새 세션에서 이어가기
  "어디까지 했었지?" →  메모리 + git 상태로 모델이 자동 재구성
```

저장될 메모의 예시:

```markdown
---
name: User auth 리팩토링 진행 상황
description: 5월 18일 시작, JWT → session 기반 전환
type: project
---

## 완료
- AuthFilter 제거 (src/main/.../SecurityConfig.kt:42)
- SessionAuthenticationProvider 추가

## 진행 중
- LoginController 마이그레이션 (POST /login만 완료)

## 다음
1. LogoutController 마이그레이션
2. 기존 JWT 토큰 호환 모드 (2주간 유지)
3. 통합 테스트 갱신
```

#### 권장 분할 단위

큰 작업은 처음부터 작은 단위로 쪼개서 각각 새 세션에서 처리하는 게 가장 안정적이다.

| 패턴 | 언제 |
|---|---|
| **한 PR = 한 세션** | PR 머지 후 다음 PR은 새 세션. 가장 자연스러운 단락 |
| **한 티켓 = 한 세션** | Jira·Linear 티켓 단위로 분리 |
| **단계별 세션 분리** | 디버깅 / 구현 / 리뷰를 각각 새 세션에서 |
| **큰 작업 분할** | 한 번에 끝낼 수 없는 작업은 단계로 나눠 각각 새 세션 |
| **70% 도달 시 정리 시작** | 95%까지 끌고 가지 말고 미리 메모 저장 + 끊기 |

#### 좋은 패턴 vs 나쁜 패턴

```
❌ 한 세션에서 며칠씩 작업
   → 압축이 누적되어 컨텍스트 품질 저하

❌ "컨텍스트 부족" 경고를 무시하고 계속
   → 비용 폭증, 모델 응답 품질 저하

❌ 진행 상태를 메모리에 저장하지 않고 새 세션 시작
   → 새 세션에서 "어디까지 했지?" 재탐색에 시간 낭비

✅ 작업 시작 시 TodoWrite로 할 일 명시
   → 진행 상태가 외부 가시화됨

✅ 컨텍스트 70% 도달 시 능동적으로 정리
   → MEMORY.md에 진행 상황 저장

✅ 자연스러운 단락(PR 머지, 단계 완료)마다 세션 끊기
   → 캐시도 살아 있고, 압축도 안 일어남
```

### 200K 임계치 관리

```
입력 토큰
   │
  ─┼─ 200K ────────────────── ← 넘으면 단가 2배
   │
   │  큰 파일은 한 번에 다 읽지 말고
   │  필요한 부분만 offset/limit으로 잘라 읽기
   │
   │  반복 탐색은 서브에이전트에 위임
   │  → 메인 컨텍스트에 결과 요약만 들어옴
   │
   ▼
```

## 모델별 비용 차이
---
사용 모델에 따라 베이스 단가가 다르다. 작업 무게에 맞춘 선택이 1차 비용 최적화다.

| 모델 | 베이스 단가 (대략) | 권장 용도 |
|---|---|---|
| `claude-haiku-4-5` | 최저 | 파일 탐색, 키워드 검색, 요약 |
| `claude-sonnet-4-6` | 중간 | 일반 구현, 코드 리뷰 |
| `claude-opus-4-7` | 최고 | 아키텍처 결정, 복잡 추론 |

서브에이전트는 메인 모델과 다른 모델을 쓸 수 있다. 검색·탐색용 서브에이전트는 `haiku`로 돌리는 게 비용 효율적이다.

```yaml
---
name: file-finder
model: haiku
tools: Read, Grep, Glob
---

당신은 파일 탐색 전문가입니다...
```

## 한 줄 요약
---
> **5분(캐시 TTL) · 200K(가격 2배) · 95%(자동 압축) — 이 세 임계치를 넘는 순간 토큰 소모량이 계단식으로 폭증한다. 베스트 케이스(0.1x)와 워스트 케이스(2.5x + 압축 비용) 사이의 격차는 25배에 달하므로, 세션 길이·휴지 시간·컨텍스트 사용량 셋을 동시에 의식해야 한다.**
