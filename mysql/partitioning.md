# [MySQL] 파티셔닝

## 파티셔닝?

---

### 1. 파티셔닝이란

파티셔닝은 하나의 테이블을 두고, 여러 개의 파티션을 나누어 특정 기준으로 데이터를 적재하는 방법이다.

이를 이용해서 데이터의 접근 범위를 full -> range로 줄일 수 있기에 탐색 범위가 비약적으로 줄어들게 된다.

![](./img/partitioning/partitioning_1.png)

파티션을 구성하게 되면, 논리적으로는 하나의 테이블이나 물리적으로 테이블 당 파티션이 생기고 각 독립적인 로컬 인덱스를 가지게 된다.

따라서 MySQL에선 파티션 전체를 아우르는 글로벌 인덱스가 아닌 파티션 독립적인 로컬 인덱스로 관리된다.

### 2. 사용 목적

앞서 그림처럼 많은 데이터를 range 별로 나눔으로써 검색의 범위를 줄이려는 목적이 있다.

실질적으로는 테이블의 행은 갯수 제한이 없기 때문에 수 많은 데이터가 적재될 수 있다. 때문에 데이터를 전체 접근한다는 관점에서 봤을 때 검색 범위가 어마어마하다. 하지만 파티셔닝을 적용하게 된다면 전체 데이터 / 파티셔닝 갯수로 관리되기 때문에 검색 범위를 좁힐 수 있다.

또한(사실 그럴 일은 없겠지만), 특정 기준으로 데이터를 모아둔 것이기 때문에 오래되거나 불필요한 데이터가 있는 파티션은 정리할 수 있다는 점도 존재한다.

그리고 인덱스는 파티션 단위로 관리되게 되는데, 인덱스의 크기 또한 작아지기 때문에 inno buffer pool에 올라갈 수 있는 인덱스 크기가 작아지게 된다.

## 파티셔닝 기준

---

파티셔닝은 다양한 기준으로 Key를 정할 수 있다.

### 1. Range

```sql
partition by range (year(created_at)) (
	partition p2024 values less than (2025),
    partition p2025 values less than (2026)
)

---

// when 구문처럼 처음 조건으로 먼저 빠지므로 2024가 p2025로 갈 일은 없음
when {
	year(created_at) < 2025 -> p2024
    year(created_at) < 2026 -> p2025
}
```

특정 범위로 파티션을 나누고, 데이터를 분배할 수 있다. 날짜, 숫자와 같은 기준이 사용되며, 위의 DDL은 연도를 기준으로 파티션을 분배한 것이다.

### 2. List

```sql
partition by list (region) (
	partition p_seoul values in ('SEOUL'),
    partition p_busan values in ('BUSAN')
);
```

고정된 카테고리로 데이터를 분배한 경우이다. 위의 DDL은 region 컬럼의 값에 따라 분배된 것이다.

### 3. Hash

```sql
partition by hash(user_id) partitions 8
```

N(DDL 예시는 8)개 중 하나의 파티션으로 user_id를 해시 함수의 결과에 따라 분배하는 방식이다. 결과적으로 나머지 연산이기 때문에 다른 방식들보다 균등하게 분배가 되기 때문에 핫 파티션 문제가 발생하지 않는다.

### 4. Key

```sql
partition by key(user_id) partitions 8
```

N(DDL 예시는 8)개 중 하나의 파티션으로 user_id를 마찬가지로 해시 함수의 결과에 따라 분배하는 방식이다. Hash 예제와의 차이점은 key인데, 이 key는 PK, Unique Key로 지정된 컬럼에만 사용할 수 있다.

## 파티셔닝 유의사항

---

### 1. 검색

테이블에 insert, select, update, delete 등과 같은 DML을 수행하더라도 일반적인 DML 쿼리를 수행하는 것처럼 사용한다. 그리고 파티션의 키가 쿼리에 포함된다면, 파티션 프루닝이 적용되어 필요한 파티션을 대상으로만 검색을 수행한다.

즉, 파티션 단위로 독립적인 인덱스가 운용되고 있으므로 테이블 전체 인덱스보다 빠르게 데이터를 탐색하고, 메모리에 상대적으로 적은 양의 인덱스를 올릴 수 있다는 장점이 존재한다.

하지만 조회 조건에 파티션 키가 없다면 어떤 파티션을 정해야 할지 모르므로 전체 파티션을 대상으로 쿼리가 수행된다. 이 경우에는 전체 파티션을 순회하면서 데이터를 조회하고, 병합을 해서 결과를 가져오기 때문에 차라리 테이블 전체 인덱스가 나은 결과를 발생시킨다.

### 2. auto increment

보통 테이블엔 auto_increment primary key를 지정하여 PK를 자동증분시키는 옵션을 이용한다. 하지만 파티션이 적용되었을 경우엔 각 파티션 별로 독립적이기 때문에 파티션 간 유니크 보장이 불가하다.

때문에 단독으로 auto_increment를 사용할 수 없고, PK를 복합키(id, created_at)와 같은 형태로 묶어서 사용할 수 있다.

혹은 PK를 지정할 때 UUID와 같은 문자열로 유니크한 값을 PK로 사용할 수 있다.

### 3. 파티션 추가, 제거

파티션은 alter DDL을 통해서 추가하고 제거할 수 있다. 상대적으로 ADD, DROP로 파티션을 추가, 제거하는 경우 테이블 구조 변경이 발생하므로 테이블 메타데이터 락이 발생하게 된다.

하지만 가령 연단위로 파티셔닝을 했다가, 월단위로 파티셔닝을 변경할 수 도 있다. 이 때는 REORGANIZATION을 수행하게 되는데, 이 때는 전체를 뒤집고 새로 재구성을 해야하는 행위이기 때문에 실 서비스에서 사용할 경우 큰 문제가 발생할 수 있다. (서비스를 닫아놓고 해야 함)

## 파티셔닝 테스트

---

### 1. 목표

MySQL의 파티셔닝 예제를 적용해보고, 애플리케이션에서 파티셔닝이 올바르게 동작하는지 확인한다.

### 2. 테스트 대상

- 파티셔닝이 적용되지 않은 상품 데이터 1,000만건
- 파티셔닝이 적용된 상품 데이터 1,000만건
- 파티셔닝이 적용된 상품 데이터의 댓글 2,000만건

위의 데이터를 이용하여 다음의 4가지 API를 구성하고, 테스트를 진행한다.

1. 파티션이 적용되지 않은 상품 데이터 조회 API
2. 파티션이 적용된 상품 데이터 조회 API
3. 상품 데이터 JOIN 댓글 쿼리에 파티션 키가 포함되지 않은 API
4. 상품 데이터 JOIN 댓글 쿼리에 파티션 키가 포함된 API

### 3. 테스트 사항

아래의 explain 예제와 같이 테이블 파티션이 적용되었고, 파티셔닝이 적용되었을 경우 어느 만큼의 성능을 확보할 수 있을지 테스트한다.

```sql
mysql> explain select * from product_partitioned where id = 1 and created_at = now();
+----+-------------+---------------------+----------------------------------------------------+------+---------------+---------+---------+-------+------+----------+-------------+
| id | select_type | table               | partitions                                         | type | possible_keys | key     | key_len | ref   | rows | filtered | Extra       |
+----+-------------+---------------------+----------------------------------------------------+------+---------------+---------+---------+-------+------+----------+-------------+
|  1 | SIMPLE      | product_partitioned | p2020,p2021,p2022,p2023,p2024,p2025,p2026,p_future | ref  | PRIMARY       | PRIMARY | 8       | const |    1 |    10.00 | Using where |
+----+-------------+---------------------+----------------------------------------------------+------+---------------+---------+---------+-------+------+----------+----------
```

API 1, 2 동시유저 50명이 2분간, API 3, 4는 동시유저 30명이 2분간 테스트를 진행하고 TPS, 평균 Latency를 측정한다.

기대하는 예상 결과는 다음과 같습니다.

- TPS: API 2 > API 1, API 4 > API 3

- Latency: API 1 > API 2, API 3 > API 4

### 4. 테스트 결과

| API | 설명 | TPS (req/s) | 평균 Latency | P95 | P90 |
| --- | --- | --- | --- | --- | --- |
| API 1 | 파티션이 적용되지 않은 상품 데이터 조회 | 16.9 | 2.13s | 4.02s | 3.69s |
| API 2 | 파티션이 적용된 상품 데이터 조회 | 29.3 | 1.18s | 2.16s | 2.00s |
| API 3 | 상품 JOIN 댓글 쿼리에 파티션 키가 포함되지 않은 조회 | 28.3 | 600ms | 1.03s | 950ms |
| API 4 | 상품 JOIN 댓글 쿼리에 파티션 키가 포함된 조회 | 28.0 | 608ms | 1.17s | 1.01s |

JOIN이 걸려있지 않은 상품 데이터 조회의 경우 기대값이 충족하나 JOIN이 걸려있는 상품 데이터 조회의 경우(쿼리가 잘못되었는지 다시 확인 필요) 기대값에 충족하지 않았다.
