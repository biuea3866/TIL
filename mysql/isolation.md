# 트랜잭션 격리 수준

## 트랜잭션
---
트랜잭션은 명령어들의 집합을 논리적 작업 단위로 묶어 원자성을 보장해주는 개념이다. 가령 A -> B에게 10,000원을 송금할 경우 A의 계좌에선 10,000원이 차감되고, B의 계좌엔 10,000원이 증가한다. 여기서 트랜잭션은 2가지의 명령을 하나의 논리적 작업 단위로 묶어 하나가 실패하면 전체를 실패시키고, 전체가 성공하면 전부 성공을 보장하게 한다. 때문에 트랜잭션은 다음의 특성을 가진다.

* Atomicity(원자성): All or Nothing. 하나가 실패하면 전체 롤백, 전체가 성공할 경우 성공
* Consistency(일관성): 트랜잭션 전후 무결성 유지
* Isolation(격리성): 동시에 실행되는 트랜잭션은 서로 간섭하지 않음
* Durability(지속성): 트랜잭션이 종료된 결과는 장애가 나도 사라지지 않음

단일의 트랜잭션은 간섭이 없기 때문에 하나의 논리적 작업만 정상적으로 수행하면 된다. 하지만 복수의 트랜잭션이 동작하고, 내부에 명령어들이 동일한 데이터를 대상으로 조회 쓰기 등의 작업이 발생하면 어떻게 될까?

데이터에 대해 락을 걸 수도 있고, 락 없이 인해 커밋된 데이터만 접근한다던지 등 여러 방법이 존재한다.

## 격리 수준
---
격리 수준은 이런 복수의 트랜잭션이 동일한 데이터를 접근할 경우 데이터를 어떻게 격리시키고, 어느 수준의 격리성으로 데이터를 보여줄 것인가에 대한 방법이다.

### 0. 동시성 이상 현상
복수의 트랜잭션이 접근하기 때문에 동시 접근 시 이상 현상이 존재한다.

* Dirty Read: 다른 트랜잭션이 커밋하지 않은 데이터(아직 작업중인)를 읽음
* Non-Repeatable Read: 같은 행을 2번 읽었는데, 값이 달라짐 (중간에 다른 트랜잭션이 업데이트)
* Phantom Read: 같은 조건으로 2번 조회했는데, 행 갯수가 달라짐 (중간에 다른 트랜잭션이 insert/delete)

### 1. READ UNCOMMITTED
커밋하지 않은 데이터도 접근 가능한 수준이다. 데이터 접근에 있어 별 다른 제약사항이 없기 때문에 성능면에선 가장 좋지만, 데이터에 대한 신뢰성이 떨어진다.

![image](./img/read_uncommited(1).png)
1. T1 트랜잭션에서 Lee 데이터를 insert

![image](./img/read_uncommited(2).png)
2. T2 트랜잭션에서 id = 51로 조회
3. T1 트랜잭션 롤백

3번으로 인해 id = 51인 데이터는 db에 존재하지 않는 데이터가 된다. 때문에 T2에서 해당 데이터를 이용하여 비즈니스 로직을 수행한다면, 잘못된 결과를 발생시킬 수 있다.

커밋되지 않은 데이터를 읽는 것을 Dirty  Read 현상이라하며, 이를 해결하기 위해 Read Committed의 격리 수준을 이용할 수 있다.

### 2. READ COMMITTED
커밋된 데이터만 읽는 격리수준이다. 커밋된 데이터만 읽기 때문에 다른 트랜잭션이 데이터를 cud 작업을 수행할 경우 undo log라는 곳에 이전 상태로 데이터를 보관한다. 그리고 다른 트랜잭션이 접근할 경우 undo log에서 데이터를 읽는다.

![image](./img/read_commited(1).png)
1. T1 트랜잭션에서 51번 id를 Jang으로 업데이트
2. 다른 트랜잭션이 접근할 수 있기 때문에 undo log에 이전 상태의 데이터를 적재

![image](./img/read_commited(2).png)
3. T2가 51번 id 접근 시 undo log에서 데이터를 조회하여 반환

이런 방법으로 Dirty Read 문제를 해소할 수 있다. 하지만 만약 T1이 commit을 하고, T2가 다시 51번 id를 접근할 경우 어떻게 될까? 이 상황에선 Non-Repeatable Read 문제가 발생한다. 

커밋된 데이터이기 때문에 T2는 같은 쿼리일지라도 다른 결과값을 받을 수 있다. 

> Undo log 플러시

undo log의 목적은 이전 상태를 보관하는 목적을 가진다. 그렇다면 커밋된 순간부터는 불필요하기 때문에 바로 플러시가 될까?

꼭 그렇지만은 않다. 이후 살펴 볼 Repeatable Read 등에서 사용되기 위해 바로 플러시는 되지 않는다. 플러시가 되는 시점은 더 이상 해당 데이터 참조가 되지 않을 때 purge thread(백그라운드 스레드)에 의해 정리된다.

### 3. REPEATABLE READ
Repeatable Read는 일관성있게 데이터를 조회하는 격리 수준이다. 즉, 몇 번이고 동일 데이터에 접근할지라도 한 트랜잭션 내에선 일관된 값을 얻을 수 있다.

![image](./img/repeatable_read(1).png)
1. id = 51인 행을 Jang으로 업데이트하고, undo log에 이전 상태를 적재한다.
2. T2는 51인 행을 undo log에서 조회한다.

![image](./img/repeatable_read(2).png)
3. T1 commit을 수행한다.
4. Repeatable Read에선 트랜잭션이 계속해서 참조하므로 undo log가 purge되지 않음.
5. 그리고 txId(version)인 스냅샷 데이터를 참조하여 일관된 데이터 조회

이렇게 mvcc를 이용하여 phantom read 현상을 방지할 수 있다. 해당 경우엔 행을 특정하여 일관된 읽기를 보장하는 경우였다. 

하지만 특정 범위에서의 N개의 데이터를 읽게 된다면, insert/delete 시 조회되는 결과가 달라질 수 있다. 이는 갭락을 이용해서 방지할 수 도 있다.

![image](./img/repeatable_read(3).png)
1. id >= 51로 N개의 데이터를 조회(select for update)하고, 결과는 1개이다.
2. 51 이후에는 아직 존재하지 않는 행에 대해서도 **갭(Gap)** 이 생기는데, `select for update`로 인해 51번 레코드 락과 함께 51 이후 구간 전체에 Next-Key Lock(Record Lock + Gap Lock)이 걸린다.

![image](./img/repeatable_read(4).png)
3. T2가 id = 52인 행을 insert 시도한다.
4. 하지만 T1이 걸어둔 갭락으로 인해 T2의 insert는 대기 상태가 된다.
5. T1이 commit 또는 rollback으로 트랜잭션을 종료해야만 갭락이 해제되고, T2의 insert가 수행된다.

MVCC 스냅샷뿐 아니라 갭락(Gap Lock) 을 통해 아직 존재하지 않는 행의 구간까지 잠금으로써 Phantom Read를 차단한다. 때문에 MySQL의 Repeatable Read는 표준 SQL과 다르게 Phantom Read까지 사실상 방지할 수 있다.

### 4. SERIALIZABLE
트랜잭션을 순차로 실행시킨다. 그리고 조회마저 읽기 잠금을 걸어 넥스트 키 락이 발생하여 cud가 불가하다.

## JPA 1차 캐시와 격리 수준
---
JPA의 영속성 컨텍스트는 트랜잭션 범위 내에서 엔티티를 캐싱하는 1차 캐시를 제공한다. 같은 트랜잭션 안에서 동일 PK로 조회할 경우 DB까지 가지 않고 1차 캐시에서 엔티티를 반환한다.

이로 인해 DB 격리 수준과는 별개로, 애플리케이션 레벨에서 일관된 읽기(Repeatable Read와 유사한 효과) 가 발생한다.

```
┌─────────────────────────────────┐
│     Persistence Context (1차 캐시)│
│  ┌───────────────────────────┐  │
│  │ Account(id=1, balance=1000)│  │ ← 최초 조회 시 적재
│  └───────────────────────────┘  │
└────────────┬────────────────────┘
             │
             ▼
        ┌─────────┐
        │   DB    │
        └─────────┘
```

### Read Committed에서의 1차 캐시
DB 격리 수준이 Read Committed이면 같은 트랜잭션 내에서도 같은 쿼리가 다른 결과를 낼 수 있다 (Non-Repeatable Read 발생 가능). 하지만 JPA 1차 캐시가 있을 경우 동일 PK 조회는 DB로 재조회되지 않고 캐시에서 반환된다.

```
T1: em.find(Account.class, 1) → DB 조회 → balance=1000, 1차 캐시에 적재
T2: UPDATE account SET balance = 500 WHERE id=1; COMMIT;
T1: em.find(Account.class, 1) → 1차 캐시 HIT → balance=1000 (DB 안 감)
```

즉, DB는 Read Committed일지라도 JPA 레벨에선 Repeatable Read처럼 보이게 된다. 단, 아래의 경우엔 1차 캐시를 우회하므로 주의해야 한다.

* **JPQL/QueryDSL 쿼리**: PK 기반 조회가 아니므로 항상 DB에 쿼리를 날린다. (조회 결과로 얻은 엔티티가 이미 1차 캐시에 있다면 캐시 인스턴스로 치환됨 — snapshot 불일치 가능)
* **네이티브 쿼리**: 1차 캐시를 무시하고 DB에 직접 질의한다.
* `em.refresh(entity)`: 명시적으로 DB 최신 값으로 덮어쓴다.
* `em.clear()` / 트랜잭션 종료: 1차 캐시 초기화
