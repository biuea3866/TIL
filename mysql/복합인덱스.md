# [MySQL] 복합 컬럼 인덱스

## 복합 컬럼 인덱스

복합 컬럼 인덱스는 두 개 이상의 컬럼을 조합하여 생성하는 인덱스다. 단일 컬럼 인덱스와 달리 여러 조건을 함께 사용하는 쿼리에서 효과적으로 동작한다.

```sql
CREATE INDEX idx_name_age ON users (name, age);
```

### B+Tree에서의 복합 인덱스 구조

B+Tree에서 복합 인덱스는 다음과 같이 정렬되어 있다. `INDEX (name, age)` 인덱스를 기준으로 다음 데이터가 있다고 가정한다.

```
users 테이블 원본 데이터:
+----+------+-----+--------+
| PK | name | age | city   |
+----+------+-----+--------+
|  1 | Park |  33 | Seoul  |
|  2 | Kim  |  25 | Busan  |
|  3 | Lee  |  28 | Seoul  |
|  4 | Cho  |  22 | Daegu  |
|  5 | Kim  |  20 | Seoul  |
|  6 | Lee  |  40 | Busan  |
|  7 | Cho  |  35 | Seoul  |
|  8 | Kim  |  30 | Daegu  |
|  9 | Park |  22 | Busan  |
+----+------+-----+--------+
```

B+Tree는 **(name, age)** 순서로 정렬한다. 먼저 `name`으로 정렬하고, `name`이 같으면 `age`로 정렬한다. 이 정렬된 결과가 리프 노드에 저장된다.

```
                         [Root]
                        (Kim,25)
                       /        \
                      /          \
              (Cho,35)            (Lee,40)
              /      \            /      \
             /        \          /        \
          Leaf1     Leaf2     Leaf3     Leaf4
```

리프 노드에는 **(인덱스 키, PK)** 쌍이 정렬되어 저장되고, 리프끼리 링크드 리스트로 연결된다. InnoDB의 세컨더리 인덱스이므로 리프 노드가 가리키는 값은 물리적 주소가 아닌 PK다.

```
  Leaf1               Leaf2               Leaf3               Leaf4
+--------------+    +--------------+    +--------------+    +--------------+
| Cho,22 →PK=4 |    | Kim,20 →PK=5 |    | Lee,28 →PK=3 |    | Park,22→PK=9 |
| Cho,35 →PK=7 |    | Kim,25 →PK=2 |    | Lee,40 →PK=6 |    | Park,33→PK=1 |
|               |    | Kim,30 →PK=8 |    |               |    |               |
+--------------+    +--------------+    +--------------+    +--------------+
       |-------->------------|-------->------------|-------->|
                       linked list
```


```
정렬된 리프 노드 (왼쪽 → 오른쪽):

  (Cho,22) → (Cho,35) → (Kim,20) → (Kim,25) → (Kim,30) → (Lee,28) → (Lee,40) → (Park,22) → (Park,33)
   ──────    ──────     ──────     ──────     ──────     ──────     ──────      ───────     ───────
   1차 정렬: name 기준 오름차순
   2차 정렬: 같은 name 내에서 age 기준 오름차순
```

첫째, **name은 전체적으로 정렬되어 있다.** `Cho → Kim → Lee → Park` 순서로 정렬되어 있어 `WHERE name = 'Kim'`이면 `Kim`의 시작 위치를 빠르게 찾아갈 수 있다.

둘째, **age는 같은 name 안에서만 정렬되어 있다.** `Kim` 안에서는 `20 → 25 → 30`으로 정렬되지만, 전체적으로 보면 `Cho의 22 → Cho의 35 → Kim의 20`처럼 age가 뒤죽박죽이다. 그래서 `WHERE age = 25`만으로는 인덱스를 탈 수 없다. age가 전체적으로 정렬되어 있지 않기 때문에 어디서부터 탐색해야 할지 알 수 없다.


이것이 복합 인덱스에서 선행 컬럼이 반드시 필요한 이유다. B+Tree의 정렬 구조 자체가 첫 번째 컬럼을 기준으로 만들어지기 때문이다.

## 선행 컬럼

복합 인덱스에서 가장 왼쪽에 위치한 컬럼을 **선행 컬럼** 이라 한다. 인덱스에 정의된 컬럼 순서대로 정렬되기 때문에, 선행 컬럼이 조건에 포함되지 않으면 인덱스를 활용할 수 없다.

### 최좌선 접두사 규칙

이를 **최좌선 접두사 규칙(Leftmost Prefix Rule)** 이라 한다. `INDEX (name, age, city)` 인덱스가 있을 때, 쿼리의 WHERE 조건에 따라 인덱스 사용 여부가 달라진다.

| 쿼리 조건 | 인덱스 사용 여부 |
| :--- | :--- |
| `WHERE name = 'Kim'` | O (선행 컬럼 사용) |
| `WHERE name = 'Kim' AND age = 25` | O (선행 컬럼 + 두 번째 컬럼) |
| `WHERE name = 'Kim' AND age = 25 AND city = 'Seoul'` | O (모든 컬럼 사용) |
| `WHERE age = 25` | X (선행 컬럼 누락) |
| `WHERE age = 25 AND city = 'Seoul'` | X (선행 컬럼 누락) |
| `WHERE name = 'Kim' AND city = 'Seoul'` | △ (`name`만 인덱스 사용, `city`는 필터링) |

선행 컬럼인 `name`이 조건에 포함되어야 인덱스를 탈 수 있다. `age`나 `city`만으로는 B+Tree의 정렬 순서를 활용할 수 없기 때문이다. 마지막 케이스처럼 중간 컬럼이 빠지면, 선행 컬럼까지만 인덱스를 사용하고 나머지는 필터링으로 처리된다.

### 인덱스 스킵 스캔

MySQL 8.0.13부터 도입된 **인덱스 스킵 스캔(Index Skip Scan)** 은 최좌선 접두사 규칙의 예외를 만들어주는 최적화 기법이다. 선행 컬럼이 WHERE 조건에 없더라도, 선행 컬럼의 카디널리티가 낮으면 옵티마이저가 자동으로 선행 컬럼의 고유 값을 순회하면서 인덱스를 활용한다.

```sql
-- INDEX (gender, age)
SELECT * FROM users WHERE age = 25;
```

일반적으로는 선행 컬럼 `gender`가 없어 인덱스를 사용할 수 없다. 하지만 인덱스 스킵 스캔이 적용되면, 옵티마이저가 내부적으로 다음과 같이 변환하여 실행한다.

```sql
-- 옵티마이저가 내부적으로 변환
SELECT * FROM users WHERE gender = 'M' AND age = 25
UNION ALL
SELECT * FROM users WHERE gender = 'F' AND age = 25;
```

선행 컬럼의 고유 값이 적을수록(카디널리티가 낮을수록) 효과적이다. 반대로 선행 컬럼의 카디널리티가 높으면 순회 비용이 커져 풀 테이블 스캔보다 느려질 수 있다. EXPLAIN에서 `Extra: "Using index for skip scan"`으로 확인할 수 있다.

## 컬럼 순서 설계

복합 인덱스의 성능은 컬럼 순서에 따라 크게 달라진다. 순서를 결정할 때 고려해야 할 핵심 기준이 있다.

### 동등 조건 우선 배치

동등 조건(`=`)으로 자주 조회되는 컬럼을 앞에 배치해야 한다. 범위 조건(`>`, `<`, `BETWEEN`)이 걸리면 그 뒤의 컬럼은 인덱스 효율이 좋지 못하기 때문이다.

```sql
-- INDEX (status, created_at)
WHERE status = 'ACTIVE' AND created_at > '2026-01-01'
-- status: 동등 조건, created_at: 범위 조건 → 둘 다 인덱스 활용

-- INDEX (created_at, status)
WHERE created_at > '2026-01-01' AND status = 'ACTIVE'
-- created_at: 범위 조건 → status는 인덱스 효율 좋지 못함
```

범위 조건은 해당 컬럼의 데이터가 전체의 몇 %냐에 따라 range scan, full scan 등이 될 수 있다.
따라서 동일한 쿼리라도 인덱스의 컬럼 순서에 따라 활용 범위가 달라진다. 범위 조건이 걸린 컬럼 이후의 컬럼들은 B+Tree의 정렬 이점을 잃게 된다.

### 카디널리티

**카디널리티(Cardinality)** 가 높은 컬럼을 앞에 배치하는 것이 일반적으로 효과적이다. 카디널리티가 높을수록 중복이 적어 인덱스로 걸러내는 데이터가 많아진다. 예를 들어, `gender`(카디널리티 낮음)보다 `user_id`(카디널리티 높음)를 앞에 배치하는 것이 검색 범위를 빠르게 좁힐 수 있다.

### 정렬과 그룹핑

인덱스 컬럼 순서와 `ORDER BY` 순서가 일치하면 별도의 정렬(filesort) 없이 인덱스 순서를 그대로 활용할 수 있다. `GROUP BY` 역시 마찬가지다. 이 점을 고려하면 쿼리 패턴에 맞춰 인덱스 컬럼 순서를 설계하는 것이 불필요한 정렬 비용을 줄이는 데 도움이 된다.

## 커버링 인덱스

**커버링 인덱스(Covering Index)** 는 쿼리가 필요로 하는 모든 컬럼이 인덱스에 포함되어 있어, 실제 데이터 페이지에 접근하지 않고 인덱스만으로 결과를 반환하는 경우를 말한다.

```sql
-- INDEX (name, age)
SELECT name, age FROM users WHERE name = 'Kim';
-- 인덱스 리프 노드에 name, age 모두 있으므로 테이블 접근 불필요
-- EXPLAIN에서 Extra: "Using index"로 확인 가능
```

세컨더리 인덱스는 리프 노드에서 PK를 찾고, 그 PK로 클러스터링 인덱스를 다시 탐색하여 실제 데이터에 접근하는 **Key Lookup** 이 발생한다. 커버링 인덱스는 이 테이블 랜덤 I/O를 제거할 수 있어 성능이 크게 향상된다. 복합 컬럼 인덱스를 설계할 때, 자주 조회하는 SELECT 컬럼까지 인덱스에 포함시키면 커버링 인덱스의 이점을 누릴 수 있다.
