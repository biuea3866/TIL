# 메시지 큐

메시지 큐는 분산 시스템에서 시스템 간 데이터를 통신하는 방법이다.
직접적인 방법으로 http 통신, grpc 통신 등의 방법이 있고, 메시지 큐는 간접적으로 시스템 간 통신을 지원하는 방식이다.
시스템 간에 메시지 큐를 두고, 메시지를 전송하여 데이터를 처리한다.

- 비동기 통신: 메시지 발행자는 수신자가 메시지를 잘 수신했는지 등에 대한 기대 결과 필요없이 메시지 큐에 메시지를 전송하고, 자신의 일을 수행할 수 있으므로 비동기적으로 동작한다.
- 느슨한 결합: 시스템은 서로 직접적인 호출이 없고, 메시지 큐에 메시지를 전송하고 읽어들임으로써 통신을 한다. 이는 서로의 존재를 몰라도 되며, 메시지 큐만을 의존하면 되므로 시스템 간의 결합도를 낮추게 된다.
- 확장성: 새로운 시스템이 메시지 큐를 구독함으로써 자신의 비즈니스를 구성할 수 있다.

# 메시지 큐 종류

## kafka

발행 - 구독 패턴을 가진 메시지 큐이다.
카프카는 컨슈머 그룹이라는 특성을 이용하여 서로 다른 시스템에서 특정 메시지 큐를 구독하여, 자신이 원하는 시스템을 확장시킬 수 있다.
그리고 이런 확장성 덕분에 분산 환경 시스템을 구축하는데 아주 유용하게 활용된다.

카프카의 메시지 큐는 토픽이라 불리우며, 시스템들은 컨슈머 그룹을 이용하여 토픽을 구독할 수 있다.
토픽을 구독 후 외부에서 시스템으로 메시지를 밀어넣는 방식이 아니라 시스템이 처리 가능한 양의 메시지를 가져오는 방식이기 때문에 백프레셔의 특징 또한 가지고 있다.

즉, 대량의 메시지가 발행되더라도 구독자 입장에서는 자신이 처리가능한 수준으로 메시지를 처리하기 때문에 시스템이 망가지진 않는다.
다만 메시지가 계속해서 큐에 쌓인 상태로 지연(lag)이 발생하기 때문에 이에 대한 해결책도 필요하다.

## rabbitmq

비동기 처리, 메시지 큐잉 등에 대한 기능을 제공하는 메시지 큐이다.
exchange를 이용하여 메시지를 라우팅할 수 있으며, queue를 exchange에 바인딩하면 fanout되어 등록된 queue에 메시지를 전달할 수 있다.

카프카는 디스크에 메시지를 저장하여, 일정 기간 동안 메시지를 저장할 수 있고, rabbitmq 또한 일정 기간 동안 메시지를 저장할 수도 혹은 재시작 시 메시지를 전부 지우는 옵션이 있다.

그리고 rabbitmq는 메시지를 pull하거나 push할 수 있는데 pull이라 함은 카프카처럼 리스너가 큐를 구독하여 시스템이 원하는 시점에 메시지를 가져가는 방식이다.
이는 지속적으로 리스너가 큐에게 polling 요청을 하는 방식이므로 지속적인 네트워크 통신을 해야한다.
다만 push 방식은 현재 리스너가 메시지를 처리 불가한 상황임에도 계속해서 메시지를 밀어넣는 것이므로 재처리를 어떻게할 것인지를 고려해야한다.

## activemq

자바 기반의 메시지 브로커이다.
그리고 JMS(Java Message Service) 클라이언트가 이 메시지 브로커를 이용하여 비동기 방식으로 시스템 간 메시지를 주고 받을 수 있다.

큐와 토픽 2개의 방식으로 브로커를 운영할 수 있으며, 큐는 구독중인 클라이언트 하나에게 전달하고, 토픽은 구독중인 여러 클라이언트에게 전달하는 방식이다.

그리고 JMS 클라이언트는 JMS 트랜잭션을 사용할 수 있어 activemq를 이용한다면 분산 트랜잭션을 사용할 수 있다.

## zeromq

별도 메시지 브로커가 없이 동작 가능하다.
노드끼리 바인딩되어 메시지를 큐잉할 수 있고, fanout되어 메시지를 전달할 수 있다.

브로커가 없기 때문에 중간에서 메시지를 저장한다던지, 트랜잭션이라던지 등과 같은 문제는 애플리케이션에서 직접 구현해야하지만 중간 노드가 없기 때문에 처치량이 매우 높으며, 지연 시간 또한 거의 없는 수준이다.

## sqs/sns

sqs는 pull 기반의 큐이고, 발행자와 구독자는 오로지 1대1 관계만을 가진다.
sqs에는 여러 기능이 존재하는데 그 중 dlq라는 기능이 있고, 이 dlq는 요청에 실패한 메시지들을 모아 재처리하는 기능을 수행한다.
카프카 등과 같은 큐는 커스텀으로 dlq를 구성하고, 실패 메시지들은 이쪽으로 흐르게 설정해주어야 한다.
다만 메시지 최대 크기가 1mb라는 점에서 id payload -> back call 등과 같은 방식 등을 이용할 필요가 있다.

sqs가 1대1 관계를 가지기 때문에 pub / sub 처럼 이벤트 브로커 형태를 가지기 위해서는 SNS를 이용한다.
SNS는 일종의 토픽이고, 발행자가 SNS로 메시지를 보내면 이에 바인딩된 SQS로 메시지가 전송되어 각 구독자들은 분산 환경으로 시스템을 구성할 수 있다.

SNS의 장점은 aws 클라우드의 다양한 시스템(S3, email, lambda)을 연동하여 처리할 수 있다는 장점이 있다.

# 현재 서비스에서 사용중인 메시지 큐

### rabbitmq

rabbitmq는 과거부터 사용해왔던 메시지 큐이지만 최근에 전부 걷어내었다.
exchange와 queue를 바인딩하여 fanout 구조를 가져가면 이벤트 브로커 형태로 kafka와 비슷하게 사용할 수 있었겠지만 결국엔 분산 환경을 구성하기 위해서 매번 queue를 만들고, exchange에 바인딩해야하는 번거로움이 있다.

또한 상대적으로 rabbitmq보다는 kafka가 친숙하고, 2개를 동시에 혼용하며 비용을 소비하기 보다는 kafka하나로 사용하는 것이 좋겠다는 판단하에 rabbitmq를 걷어내게 되었다.

### activemq

분산 트랜잭션을 지원한다는 면에서 좋은 선택지(JTA Coordinator) 같아보이나 모든 로컬 트랜잭션이 커밋이 될 때까지 api가 블로킹되고, 하나라도 실패할 경우 전체 롤백 전파를 해야한다는 면에서 성능 상 좋지 않다.

그리고 push 형태로 메시지를 전달하므로 구독자 애플리케이션이 처리 불능 상태일 경우 타임아웃 에러 등에 빠질 위험이 높다.

### zeromq

성능 면에서 매우 좋으나 메시지의 영속성이 없고, 메시지 트랜잭션이 지원되지 않는다는 면에서 원자성이 중요한 웹 애플리케이션에선 부적절하다.

### sns / sqs

kafka랑 비교해서 2개를 조합했을 때 다른 queue에 비해 굉장히 좋다.
그리고 다른 aws 기능들을 sns에 다양하게 연동 가능하다는 점에서 확장성 또한 좋다.

다만, 메시지 크기가 최대 1mb일 정도로 작고, 다른 시스템에서 sns를 구독하기 위해서 별도로 queue를 생성해야한다는 면에서 번거로움이 존재하고, 리소스 관리 측면에서 힘들다고 생각한다.

# 메시지 큐 비교 요약

| 항목 | Kafka | RabbitMQ | ActiveMQ | ZeroMQ | SQS/SNS |
|------|-------|----------|----------|--------|---------|
| 방식 | Pull (poll) | Push / Pull | Push | P2P | Pull (SQS) |
| 메시지 저장 | 디스크 (보존 기간 설정) | 메모리/디스크 선택 | 디스크 | 없음 | AWS 관리형 |
| 순서 보장 | 파티션 내 보장 | 큐 단위 보장 | 큐 단위 보장 | 보장 안 됨 | FIFO 큐 사용 시 보장 |
| 처리량 | 매우 높음 | 중간 | 중간 | 최상 | 높음 |
| 확장 모델 | 컨슈머 그룹 | Exchange-Queue 바인딩 | 큐/토픽 | 직접 구현 | SNS-SQS 조합 |
| 분산 트랜잭션 | 미지원 | 미지원 | JTA 지원 | 미지원 | 미지원 |
| DLQ | 커스텀 구현 | 지원 | 지원 | 없음 | 네이티브 지원 |
| 운영 난이도 | 높음 (ZK/KRaft) | 중간 | 중간 | 낮음 | 낮음 (관리형) |

## 선택 기준

* **높은 처리량 + 이벤트 소싱**: Kafka
* **복잡한 라우팅 + 프로토콜 다양성**: RabbitMQ
* **JMS 표준 + 분산 트랜잭션**: ActiveMQ
* **초저지연 + 브로커 없는 구조**: ZeroMQ
* **AWS 생태계 + 관리 비용 최소화**: SQS/SNS
