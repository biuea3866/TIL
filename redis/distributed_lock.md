# [REDIS] 분산락

## 락

---

### 락이란?

락은 여러 수행 주체가 동시에 같은 자원(파일, 가변 변수, 메모리 등)에 접근할 때 자원에 대한 일관성을 유지하기 위해 다른 수행 주체에 대한 접근을 막기 위한 동기화 메커니즘이다. 상호 배제를 통해서 경합(Race Condition)을 방지한다. 경합은 여러 수행 주체가 동시 접근 시 A의 작업이 다른 수행 주체의 작업으로 덮어씌워질 수 있는 현상이다. 즉, 락은 특정 시점에 하나의 수행 주체만이 데이터를 쓰기 작업할 수 있도록 보장해주는 역할을 한다.

### 뮤택스, 세마포어

뮤택스와 세마포어는 동시성을 제어하는 동기화 메커니즘이다.

#### 1. 뮤택스

뮤택스는 상호 배제를 보장하는 잠금 장치로 한 번에 하나의 스레드만 임계 영역(Critical Section)에 진입할 수 있게 한다. 락을 획득한 스레드만 락을 해제할 수 있는 권한이 있다.

![](./img/dist_lock_1.png)

소유권이 존재하는 T1만이 락을 해제 가능하며, T2는 접근이 불가하다. Java에선 이를 Synchronized, ReentrantLock으로 구현하여 상호 배제를 보장한다.

#### 2. 세마포어

뮤택스가 하나의 스레드를 위한 상호 배제를 보장했다면, 세마포어는 여러 스레드가 임계 영역에 제한된 숫자만큼 동시에 진입할 수 있도록 보장해주는 장치이다. 세마포어에선 임계 영역에 도달한 수행 주체 각자가 락을 해제할 수 있다.

![](./img/dist_lock_2.png)

## 분산락

---

### 분산락이란?

뮤택스, 세마포어 등은 모두 단일 프로세스 내에서 동작화는 동기화 메커니즘이다. 즉, 멀티 인스턴스가 동작하는 분산 환경에서는 동일 자원에 대한 동시 접근 제한이 불가하다.

![](./img/dist_lock_3.png)

각 인스턴스에서 뮤택스로 상호 배제를 할지라도 데이터를 가져와 단일 프로세스 내에서만 상호 배제가 동작할 뿐이다. 데이터베이스라는 공유 접근 자원에 대한 상호 배제는 보장되지 않는다.

분산락은 이런 멀티 인스턴스들이 접근하는 공유 자원(데이터베이스)에 대한 상호 배제를 위해 다른 외부 시스템을 이용하는 방식이다. Redis, Zookeeper, MySQL 등을 이용해 분산락 구현이 가능하며, Redis가 많이 선택된다.

### 레디스의 분산락

Redis의 분산락이 많이 사용되는 이유는 원자성 보장, 메모리 기반의 빠른 연산이 이유이다. Redis는 작업 대상들이 Queue에 순차적으로 쌓이고, 싱글 스레드가 순서대로 작업을 수행하기 때문에 동시성 문제가 발생하지 않는다. 때문에 분산락 구현 시 Redis를 많이 사용한다.

하지만 이런 구현은 Redis가 단일 인스턴스일 경우에만 가능하다. Redis도 멀티 인스턴스 (Master-Slave)로 구축을 할 수 있고, 멀티 인스턴스 방식은 단일 인스턴스가 장애가 발생했을 경우 빠르게 장애 복구가 가능하도록 도와준다. 즉, 비동기 복제 방식에서의 복제 지연이 락 획득에 대한 원자성을 보장해줄 수 없을 수 있다.

![](./img/dist_lock_4.png)

1. Client A는 Master로부터 락을 획득한다.

2. Master는 Slave에게 락 정보를 복제시켜야하는데, 인스턴스가 다운된다.

3. Slave 인스턴스는 Master로 승격되지만, 앞서 Client A가 획득한 락의 정보를 모른다.

4. Client B는 신규 Master로부터 Client A와 동일한 락을 획득한다.

Master-Slave에선 이와 같은 상황이 발생할 수 있다. 따라서 이를 해결하기 위한 방법으로 RedLock 알고리즘이 생기게 되었다. RedLock은 멀티인스턴스가 구축된 클러스터에서 가능하며, N개의 인스턴스 중 과반수로부터 락을 획득하면 분산락을 획득한 것으로 판단한다. 과반수로부터 락을 획득했기 때문에 인스턴스 하나가 다운되어도 다른 인스턴스들은 락에 대한 정보를 알고있다. 즉, Client B가 락을 획득하려고 해도, 과반수로부터 락을 획득할 것이기 때문에 Master-Slave에서의 문제는 발생하지 않는다.

![](./img/dist_lock_5.png)

![](./img/dist_lock_6.png)

Instance B가 다운되어도 Instance A, D에선 락에 대한 정보를 알고 있기 때문에 Client B를 대기시킨다. (갑자기 Instance A, B, D만 골라서 다운되는 상황이 발생한다면 또 모르겠지만 그럴 일은 잘..)

이 외에도 RedLock 알고리즘 문제, 애플리케이션 중단, 네트워크 지연 등에 대한 문제로 한계점이 존재하지만 깊게 다루지는 않고, 실제 분산락이 어떻게 사용되는지 알아보기로 한다. (다른 블로그 글에 잘 정리되어있어 참고하면 좋을 것 같다.)

### 레디스 분산락 구현 (Lettuce, Redisson)

Lettuce는 멀티스레드 환경에서 Thread safe을 보장하는 redis client이다. Lettuce로 분산락을 사용할 경우 while 문을 걸고, 계속 락 획득을 시도하는 polling 형태(스핀락)로 구현이 된다. 반면에 Redisson은 pub / sub으로 락 획득을 구독하기 때문에 polling 시도를 하지 않는다. 사실 이건 lettuce를 이용하여 pub / sub 형태로 만든다면 똑같다. 다만, Redisson의 경우 많은 기능들이 구현되어있어 저수준으로 존재하는 Lettuce보다는 활용하기 편할 수 있고, 고수준으로 존재하는 Redisson의 경우 강제되는 것들(커넥션 설정, 메모리 사용 등)이 있기 때문에 선택하면 된다.

![](./img/dist_lock_7.png)

스핀락 형태의 락 획득 시도로 특정 주기를 두고 지속적으로 polling 요청을 시도한다. 지속적인 락 획득 시도는 redis 인스턴스에게 계속해서 연산을 하도록 요구하는 것이므로 cpu 자원 사용, 다른 요청 블로킹 등에 대한 문제를 야기시킬 수 있다.

![](./img/dist_lock_8.png)

pub / sub 형태로 락 획득을 시도하는 패턴이다. Client A가 이미 락을 가지고 있으면, Client B는 락 획득에 실패하게 된다. 이 때 Client B는 개설된 Channel을 구독하여 락이 해제되면, Redis로부터 알림을 받고 락을 획득하게 된다.

## 분산락 고려 케이스

---

사실 분산락을 알아본 것은 따닥 문제 때문에 리소스가 중복으로 생성되는 일이 많아져서 막으려고 공부해본 것이다. 분산락을 적용하기 위해서 wait, lease 시간을 고려해야하는데, 어떤 기준으로 초를 산정하면 좋을지 케이스들을 알아보았다.

### **lease(10) >  api 종료(4) >**wait(1)

![](./img/dist_lock_9.png)

락 대기 시간보다 api 종료가 길어 락 획득 실패 에러가 발생하는 케이스이다. 이럴 경우 에러를 캐치하여 이미 처리된 요청, 처리중인 요청중 등에 대한 에러 문구를 반환할 수도 있다.

### **lease(10) > wait(4) >**api 종료(3)

![](./img/dist_lock_1.png0)

첫 요청이 정상적으로 처리되고, 락이 반환되어 두 번째 요청은 정상적으로 처리되는 과정이다. 만약 POST API(멱등성 x)라면 중복으로 리소스가 생성될 수 있으니 껄끄러운 상황이 발생할 수 있다. (중복 주문, 중복 결제) 따라서 멱등성있는 API라면 상관없겠으나 막아야겠다라고 하면 wait을 0으로 주고, 아예 튕겨버리게 할 수도 있다.

### api 종료(6) >**lease(5) **> wait(4)

![](./img/dist_lock_1.png1)

최악이다. 비즈니스 로직이 길어져 만료 시간을 넘어버리게 된 케이스이다. 이 때는 락이 만료 시간을 지났으므로 해제가 되며, T2에선 락을 획득하게 된다. 즉, 로직을 수행할 수 있는 권한이 생긴 것이고 중복된 리소스가 생성될 수 있다.

추가적으로 Redisson에는 WatchDog라는 기능이 존재한다. Watch Dog는 주기적으로 락 만료 시간을 초기화시켜주는 기능인데, 예상치 못하게 비즈니스 로직이 오래 걸리는 경우 락이 조기에 해제되어버려 동시성 문제가 발생할 수 있는데 만료 시간을 초기화시켜버려 이런 문제를 방지해준다. WatchDog 기능을 사용하기 위해선 lease time에 -1을 주거나 값을 주지 않으면 동작하게 된다.
