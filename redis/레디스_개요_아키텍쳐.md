# [redis] 레디스 개요와 아키텍처

## Redis란
---

Redis는 메모리에 저장하고, 조회하는 인메모리 기반 key-value 데이터 저장소이다. 캐싱, 메시지 브로커, 스트리밍 등 다양한 활용이 가능하다.

Redis가 고성능, 저지연이라는 특징을 가지고 있는데 그 이유는 다음과 같다.

1. **메모리 기반 데이터 저장**: 데이터를 메모리에 저장하고, 주기적으로 스냅샷을 디스크에 저장한다.
2. **싱글 스레드 및 이벤트 루프 시스템**: 이벤트 루프와 I/O Multiplexing을 통해 TCP 연결을 빠르게 수립하고, 싱글 스레드가 메모리 위에서 빠르게 단건씩 처리하기에 Lock 없이 높은 처리량을 보장한다.

보통 초당 10만 건 이상의 명령을 처리할 수 있으며, 평균 응답은 1ms 미만이다.

## 멀티 스레드 모델의 한계

---

전통적인 멀티스레딩은 클라이언트가 요청이 들어오면, 프로세스의 스레드가 이 연결을 수락하고 처리하게 된다. 이는 요청 당 스레드 1개를 할당하는 것이고, 프로세스의 스레드 풀에 존재하는 N개의 스레드들을 할당한다. 이로 인해 멀티 스레딩 환경은 동시에 N개의 요청을 처리할 수 있다.

하지만 여기서는 문제점이 존재한다.

* **컨텍스트 스위칭**: 선점형 스케줄링에선 특정 시간 동안만 스레드가 CPU 자원을 할당받아 일을 처리하게 된다. 만약 일을 특정 시간 내에 처리하지 못하면, 다음 스케줄링으로 예정된 스레드에 CPU 자원을 할당하여 다음 스레드가 작업을 한다. 이렇게 스케줄링에 의해 스위칭되는 것을 컨텍스트 스위칭이라한다. 이 과정에선 CPU 레지스터에 스레드가 작업을 위한 스택 메모리 값이 저장되어 있는데 이 레지스터의 값을 저장하고, 재작업 시 복원해야한다. 또한 CPU 캐시도 무효화시켜야 한다. 이런 오버헤드만으로 CPU는 상당 시간을 소비하게 된다.

* **동기화 복잡성**: 여러 스레드는 메인 메모리의 공유 자원에 대해 접근하여 변수를 복사하여 작업을 처리한다. 이 변수는 CPU 캐시(독립적으로 사용되는)에 저장되기 때문에 메인 메모리에 다시 값을 업데이트한다. 이 과정에서 A 스레드가 작업한 값을 B 스레드가 덮어씌우게 되는 현상이 발생하는데, 이를 동시성 문제라하며 이를 해결하기 위해 동기화 과정이 필요하다. 멀티스레딩 환경에서 공유 자원에 대한 동시성 문제는 Lock을 걸음으로써 해결이 가능한데, 이 Lock은 다른 작업을 블로킹시키므로 성능을 생각한다면 좋은 방법은 아니다.

## Redis 아키텍처

---

### Redis가 선택한 방식

Redis는 하나의 스레드가 I/O Multiplexing을 이용하여 수 많은 연결을 감지하고, 준비된 연결을 골라 순차적으로 처리한다. 즉, 스레드 당 하나의 연결을 처리하는 게 아닌 하나의 스레드가 수 많은 연결을 처리하게 된다.

### Redis 아키텍처

![alt text](<../img/redis/레디스_개요_아키텍처/스크린샷 2026-03-28 17.18.23.png>)

1. 클라이언트와의 연결을 맺기 위해 소켓을 형성하고, 운영체제로부터 파일디스크립터를 반환받아 소켓과 매핑한다.
    * 이 파일디스크립터는 단일 스레드가 감시하며, epoll로 반환된다.
2. 소켓은 수신 버퍼 / 송신 버퍼를 가지고 있다.
3. 수신 버퍼에 데이터가 도착하면 ready list로 파일 디스크립터가 등록된다.
4. 이벤트 디스패처 -> 핸들러 테이블 -> 명령 프로세서 -> 응답 큐의 과정을 거쳐 데이터를 I/O한다.
    * 이 과정을 이벤트 루프라 부른다.
5. 응답 큐 -> 송신 버퍼로 write 함수를 이용하여 송신 버퍼에 데이터를 반환한다.
6. NIC를 이용하여 소켓의 송신 버퍼에서 데이터를 꺼내어 클라이언트에게 응답한다.

> 수신 버퍼 / 송신 버퍼

수신 버퍼는 클라이언트로부터 전달되는 네트워크 패킷들을 모으는 버퍼이다. 송신 버퍼는 응답된 결과를 쌓는 버퍼이다.

> 파일 디스크립터

OS가 열린 소켓을 식별하기 위해 부여하는 음이 아닌 정수이다. socket 함수를 호출하여 fd를 얻고, bind와 listen 함수를 거치면 클라이언트 연결마다 새로운 fd를 반환한다.

> select

fd 테이블에 존재하는 모든 fd를 순회하여 이벤트가 발생한 fd를 조회한다. (1024개로 fd 제한) epoll 이전에 사용하던 방식이며, fd를 조회하기 위해 O(N)의 시간 복잡도가 발생한다.

> poll

select의 fd 갯수 제한을 없애고, 관심 이벤트와 발생 이벤트를 분리한다. 하지만 여전히 전체 순회를 한다는 단점이 있다.

> epoll

관심 목록을 만들고 fd를 등록하여 레드 - 블랙 트리로 관리한다. fd 등록/삭제는 O(logN)이며, 이벤트가 발생한 fd만 별도의 ready list로 반환하기 때문에 이벤트 조회는 O(ready fd 수)로 전체 순회할 필요가 없다.

즉, Redis는 수 많은 요청을 I/O Multiplexing으로 연결을 수립하고, 메모리에 대해 I/O를 발생시킨다. 이로 인해 수 많은 요청을 동시에 처리할 수 있으며, 또한 빠르게 데이터 I/O할 수 있다.

### 멀티스레딩 아키텍처 (Redis 6.0+)
Redis 6.0부터는 네트워크 I/O 부분만 멀티 스레딩을 도입하여 성능을 향상시켰다. 앞서 그린 도표에서 수신 버퍼에서 데이터를 읽는 과정(read), 송신 버퍼에 데이터를 쓰는 과정을 멀티스레딩으로 병렬처리한다.

네트워크 I/O만 멀티 스레딩을 처리하는 이유는 이 과정에서 병목이 생기기 때문이다. 클라이언트와의 연결이 많아지고, CPU 코어 하나가 스레드 1개를 이용하여 계속해서 네트워크 I/O를 처리하여 리소스가 네트워크 I/O에 대부분 할애된다. 따라서 멀티스레딩으로 네트워크 I/O를 병렬(별도 I/O스레드)로 처리하고, 기존 명령 처리는 단일 스레드(메인 스레드)로 처리함으로써 역할을 나누어 명령 수행에 더 많은 리소스를 할애할 수 있게 된다.

이런 아키텍처로 변경하여 Redis는 보다 많은 성능 향상을 할 수 있게 되었다.
